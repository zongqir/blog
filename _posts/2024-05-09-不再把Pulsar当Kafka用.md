这篇文章不是一个“中间件选型安利贴”，而是一次完整的工程复盘。  
我想讲清楚一件事：我们在多租户 SaaS 数据采集场景里，为什么会从一租户一容器，走到 etcd 调度，再走到 Pulsar 原生多租户架构；每一步解决了什么问题，又引入了什么问题；最后为什么选择把复杂度收回到 Pulsar + 云原生消费者这条主线上。

先加一个修订说明：这版内容明确纠正了我之前一些“讲得太满”的表述。  
尤其是以下四点，我会在文中给出边界：

1. ACK 只能解决 Broker 游标进度，不解决计算资源隔离。
2. 正则订阅不是银弹，Topic 多时会带来 metadata 轮询与重建抖动。
3. 分桶不是“无调度”，本质上仍然是调度，只是从强动态转为弱中心、可运维的调度。
4. Kafka 不是“不行”，在本规模也能做；我们选 Pulsar 是语义匹配，不是性能神话。

如果你也在做多租户采集、消息编排、异构下游写入，而且团队里正在争论“要不要做一套自己的租户调度平台”，这篇文章可以作为一个完整对照样本。

## 一、问题定义：我们要解的不是“消费消息”，而是“多租户下的长期可演进采集”

最早业务看起来很简单：

1. 从 MQ 消费消息。
2. 调下游接口做业务编排。
3. 把结果写入数据库。

但当业务从“少数大客户”走向“多租户 SaaS”之后，问题空间变了，复杂度不在单点，而在组合：

1. 租户数量增长，接入和下线更频繁。
2. 租户流量分布极不均衡，小租户多、大租户峰值高。
3. 不同租户下游依赖差异大，耗时和失败模式不同。
4. 系统既要扛吞吐，也要保证租户隔离和公平性。

所以我们真正要解的不是“单条链路跑通”，而是以下四个目标同时成立：

1. 租户边界清晰可观测。
2. 扩缩容动作简单可预期。
3. 慢租户不能拖垮快租户。
4. 写入链路在当前规模下可控、可排障、可演进。

## 二、v1：一租户一容器，直观但很快撞墙

我们第一版非常朴素：

1. 一个租户对应一个容器（Pod）。
2. 每个容器里一个消费者进程，独立消费、编排、入库。

这版的优点很诱人：

1. 隔离天然，某租户挂了不容易波及其他租户。
2. 排障直观，一个租户一份日志一份进程状态。

但短板也非常快暴露：

1. 资源利用率低，很多长尾租户流量很小却要独占一个 Pod。
2. 配额难配，小了怕峰值击穿，大了浪费集群资源。
3. 运维复杂度随租户数量线性甚至超线性增长。
4. 升级和治理动作变成“对 N 个租户重复做 N 次”。

简单说，v1 是一个“能跑但不可规模化”的形态。

## 三、v2：etcd 分配器 + 多租户协程，认真做过，也认真放弃

在 v1 撞墙后，我们没有立刻改用 Pulsar 原生能力，而是先走了一个看起来非常合理的中间态：  
借鉴公司其他团队的成熟实践，做“分配器 + worker + etcd 协调”的租户调度模型。

### 3.1 v2 的基本结构

1. 分配器（Allocator）定期拉取租户列表和 worker 负载。
2. 计算租户到 worker 的映射，并写入 etcd。
3. worker watch etcd 前缀，感知分配变化。
4. worker 进程内部用协程承载多个租户处理逻辑。

这条路的吸引力很真实：

1. 有内部成功样本，不是从零造轮子。
2. 能把“一租户一容器”收敛到“多租户共进程”。
3. etcd 做协调在工程上是成熟方案。

### 3.2 v2 解决了什么

1. Pod 数量显著下降，资源利用率变好。
2. 租户动态增删不再靠手工操作部署。
3. 系统从“静态部署思维”迈向“动态承载思维”。

### 3.3 v2 为什么最终没成为主线

问题不在 etcd 本身，而在“它是不是这个问题空间里最合适的重心”。

第一，系统重心偏移。  
我们的主战场是多租户数据采集和写入治理，不是“打造一个租户调度平台”。  
但 v2 的复杂度会把团队精力持续拉向调度层。

第二，负载建模比想象复杂。  
只看消息条数会失真，因为消息成本差异巨大。  
一旦加入耗时、错误率、下游限流、重试代价等维度，你就在做一个 mini 调度器 + 指标系统。

第三，etcd 变成强依赖且和业务语义割裂。  
etcd 在这里承载了“租户分配真相”，一旦抖动会放大为采集系统级风险；  
但它的语义与我们最关心的租户、SLA、topic backlog 等核心业务对象并不天然对齐。

我们当时给这个结论的表述是：  
**v2 不是错误方案，但它在我们的问题空间里过重，且重心不对。**

## 四、关键转向：不是“去调度”，而是把调度复杂度分层

这个阶段我们做的不是“抛弃调度”，而是“调整调度的位置和形态”。  
之前那种“去掉 etcd 就等于不做调度”的说法并不准确。

我们真正做的是三层分工：

1. 平台层调度：Kubernetes 负责副本扩缩与节点调度。
2. 消息层路由：Pulsar 负责 subscription 分发与消费进度管理。
3. 业务层分流：我们负责租户分桶、大租户剥离、限流熔断。

也就是说，我们没有消灭调度，而是避免再自建一个“强一致控制平面”去管所有租户分配细节。

为什么这一步关键？因为我们之前虽然在用 Pulsar，但用法接近 Kafka：

1. topic 主要按业务线拆。
2. 多租户混在同一消费链路。
3. 消费提交逻辑偏顺序化思维。

在小规模阶段这样没问题，在多租户规模化阶段就会暴露：

1. 租户边界不清，排障和SLA管理困难。
2. 租户间拖累明显，慢租户影响整体吞吐。
3. 新租户接入缺乏系统级“一等公民”建模。

所以后面的架构不应理解成“无需调度”，而应理解成“把调度从重平台化收敛为分层治理”。

## 五、v3 最终架构：一租户一 topic + 正则订阅 + shared 扩缩容

### 5.1 租户提升到 topic / namespace 级别

我们把租户从消息字段抬升为消息系统层级对象：

1. 一租户一 topic（或一租户一组 topic）。
2. 统一命名约定，例如 `persistent://saas/tenant-<tenantId>/ingest-main`。

直接收益很明确：

1. 删除租户可直接回收租户相关 topic/namespace。
2. 排障可直接查看租户级 backlog、延迟、错误率。
3. 权限控制可在 Pulsar 层天然落位。

这不是“命名规范小优化”，而是架构语义变化：  
**租户是一等公民，不再是业务字段。**

### 5.2 消费者正则订阅，自动承接新增租户

消费端不再维护“租户到worker”的外部映射表，而是：

1. 通过正则订阅一组租户 topic。
2. 新租户创建 topic 后自动被消费者感知。

例如小租户可以走类似 `tenant-(?!big-).*` 的模式，配合统一命名即可自动接入。  
但这里必须把边界讲清楚：正则订阅是“降运维成本工具”，不是“零代价能力”。

在 Topic 规模较大时，正则匹配通常伴随元数据发现和 Consumer 内部 Handler 管理开销。  
一旦新 Topic 大量出现或 Topic 抖动，消费端可能出现短时抖动或重建成本。  
所以我们的实践不是“全场景无脑正则”，而是：

1. 长尾租户池可用正则自动发现。
2. 高 SLA 租户优先走显式 topic 列表或独立消费者组。
3. 通过 namespace 分片降低单次元数据扫描压力。
4. 把租户批量接入安排在可控时间窗，避免业务高峰抖动放大。

### 5.3 扩缩容方式：加消费者副本，而不是改分配规则

在 shared subscription 下，扩容动作可以简化为：

1. 增加消费者副本数。
2. 由 Pulsar 在同一 subscription 内分发消息。

也就是说，扩容不再是“调度规则升级工程”，而是“标准横向扩容动作”。  
这对稳定性和值班体验都有实质改善。

### 5.4 大小租户分流，保证公平性与 SLA

多租户系统一定有大租户和长尾租户，不能混成一锅：

1. 小租户通过一套正则订阅和共享消费者承载。
2. 大租户单独正则或显式 topic 列表，挂专用消费者池。
3. 大租户独立资源配额、监控、告警和限流策略。

这样做的核心收益是：  
平台统一，但SLA分层；重租户可重保障，长尾租户不被拖垮。

## 六、ACK 的边界：它能隔离游标进度，不能替代计算资源隔离

这部分必须纠偏。  
我之前的表达容易让人误解成“有了 per-message ACK，慢租户就不会影响快租户”。这个结论不成立。

多租户共进程场景有一个硬现实：

1. 有些租户下游很快。
2. 有些租户下游波动大、重试多。
3. 有些租户会在短时峰值下产生长 backlog。

如果提交语义过于顺序绑定，就会出现“慢租户绑架快租户”的现象。  
Pulsar 的 per-message ACK/NACK 能解决的是“Broker 端进度绑定”问题，而不是“Pod 内资源争用”问题。

ACK/NACK 的真实价值是：

1. 成功消息立即 ACK，不受其他租户失败消息拖累。
2. 失败消息单独 NACK，按策略重试或进入死信。
3. 让消费进度管理不被单条坏消息卡住。

但要防止慢租户拖垮快租户，必须再加“计算侧隔离”：

1. 按租户或租户等级设置并发上限（bulkhead）。
2. 下游调用线程池按等级隔离，避免全局线程池被打满。
3. 数据库连接池做配额，避免单租户连接吃满。
4. 每租户内存队列设置上限，触发背压、降级或延迟重试。
5. 重试预算（retry budget）按租户计，不允许无限重试吞资源。

所以我们最终能长期运行多租户共进程，不是因为 ACK 一招制胜，  
而是“游标隔离 + 计算隔离 + 资源配额 + 重试治理”四件事一起成立。

## 七、写入链路：去租户化 + 合并写入 + 当前规模下串行化

消费侧稳定之后，系统是否可长期运行，关键在写入链路能否“稳、准、可回放”。

### 7.1 去租户化统一模型

消息进入写入阶段前，先抽象为统一内部模型：

1. 租户ID。
2. 业务主键。
3. 数据载荷。
4. 合并和落库策略。

写入器关注“怎么落库”，不关注“租户业务语义细节”，这样可维护性更好。

### 7.2 合并写入减少无效IO

在内存写队列里，对“同租户 + 同主键/同类型”的短时多次更新做压缩合并，  
把高频碎写收敛成更少的有效写，降低数据库压力。

### 7.3 当前 EPS 量级下选择串行写入

我们当前整体 EPS 大约在 3000 左右。这个量级下，串行写入配合前置合并是可行的，收益是：

1. 并发控制简单，可推导性强。
2. 排障和重放逻辑更清晰。
3. 故障恢复路径更短。

这不是“永恒最优解”，而是“与当前规模匹配的工程解”。  
当 EPS 再上一个数量级，演进方向也很明确：

1. 热点租户拆分独立写入 worker。
2. DB 前增加持久化缓冲/WAL，隔离瞬时抖动。
3. 强化幂等键和回放策略，保证一致性。

## 八、trade-off 讲清楚：我们不是拿到了免费午餐

v3 方案带来的是“复杂度重排”，不是“复杂度消失”。  
我们主动接受了以下权衡：

1. 放弃自研强动态调度，换取系统重心回归业务主线。
2. 接受 topic/订阅治理复杂度，换取租户语义清晰和自动化接入。
3. 接受 ACK/NACK + 资源隔离治理工作量，换取慢租户不绑架快租户。
4. 当前写入端先用串行化，换取稳定和可排障，未来再按量级演进。

这几个 trade-off 都是显式做出的，而不是被动承受。  
同时也要明确：分桶本身就是调度策略，只是比 etcd 动态分配更“静态、更可控、运维成本更低”。

## 九、不适用边界：这套方案并不适合所有团队

为了避免“万能叙事”，边界必须写清楚。  
以下场景不建议直接照搬：

1. 你的核心产品就是调度平台本身。  
在这种场景里，自研分配器可能就是主价值，而不是负担。

2. 你有极强、跨租户、全局顺序一致性的提交要求。  
这时需要额外设计一致性策略，不能只依赖默认消费语义。

3. 你的团队尚未建立基本可观测能力。  
没有租户级指标、重试指标、死信治理，细粒度 ACK 只会让问题更隐蔽。

4. 你的业务仍在单租户早期验证。  
过早引入完整多租户模型会提高起步成本。

## 十、迁移步骤清单：从“现有系统”平滑切到 v3 的可执行路线

这部分给到可复用步骤，供你按阶段落地。

### 阶段 1：语义建模与命名统一

1. 定义租户级 topic/namespace 命名规范。
2. 梳理现有 topic 与租户映射关系。
3. 补齐租户级监控看板（backlog、延迟、错误率、重试率）。

### 阶段 2：消费端能力改造

1. 引入正则订阅能力，先覆盖长尾租户。
2. 按租户拆分处理队列和协程。
3. 建立 ACK/NACK 重试与死信策略。

### 阶段 3：容量和SLA治理

1. 大小租户分层，建立独立消费者池。
2. 通过 shared subscription 做弹性扩缩容。
3. 建立租户分级限流与熔断策略。

### 阶段 4：写入链路稳定化

1. 落地去租户化内部写模型。
2. 实施短窗合并写入，观测 DB 压力曲线。
3. 补齐幂等、回放、补偿和审计机制。

### 阶段 5：回收旧链路与发布门禁

1. 灰度迁移租户，保留可回退路径。
2. 对比新旧链路指标，验证吞吐、延迟、错误和资源占用。
3. 达到门禁后再下线旧分配逻辑。

## 十一、补充讨论：一个消费者订阅 3000 个 topic，真的能长期扛住吗？

这部分是我们在实践里反复被问到的问题，我把结论写得直白一点：

1. 从“功能上能否建立连接”看，3000 topic 并非绝对不可能。
2. 从“生产上能否长期稳定承载”看，它通常是高风险设计。
3. 真正要看的不是 topic 数字本身，而是 topic-partition 总量、预取队列、ACK 轨迹、故障域大小和公平性。

### 11.1 先看真实规模：不是 3000，而是 3000 x 分区数

很多评审会只报“topic 数量”，但消费者真实感知的是 topic-partition 维度。  
如果 3000 个 topic 每个 3 分区，实际就是 9000 条分区消费状态要维护。

这意味着：

1. 客户端状态管理成本不是线性小增长，而是显著放大。
2. rebalance、重连、恢复时的抖动范围会更大。
3. 监控面板“看起来还好”不代表恢复路径可控。

### 11.2 内存压力通常来自预取队列，而不是你直觉里的业务逻辑

在 topic 数多时，最容易被低估的是消费者端预取缓存。  
你可以用一个粗算模型先做容量预估：

`内存占用 ≈ 有效topic数 x receiverQueueSize x 平均消息体积 x 对象开销系数`

比如：

1. 有效 topic 3000。
2. `receiverQueueSize` 100。
3. 平均消息体 2KB。

光消息体理论值就是 600MB，考虑对象、索引、序列化和 GC 元数据后会更高。  
所以“一个进程吃全量 topic”的代价，常常是先死在内存和 GC，而不是先死在 CPU。

### 11.3 ACK/NACK 的优势还在，但管理成本会抬升

细粒度 ACK 是 Pulsar 在多租户场景里的关键能力，这点我们仍然坚持。  
但 topic 量级上来后，你需要承认另一面：

1. unacked 追踪与超时重投的状态管理更重。
2. 某些抖动租户会放大重试轨迹，影响消费者稳定性。
3. 需要更细的租户级重试上限、退避与死信策略。

换句话说，ACK 机制不是问题，问题在于全量 topic 混跑下的运维复杂度。

### 11.4 故障域过大是比吞吐更危险的风险

单实例订阅全量 3000 topic 的另一个大问题是故障域。  
一旦进程崩溃、发布重启或节点抖动，你会遭遇：

1. 全量租户同时受影响。
2. backlog 恢复阶段出现集体风暴。
3. 值班窗口里“同时红很多租户”，定位成本急剧上升。

所以我们在评审时会明确一句：  
**3000 topic 单消费者可以作为压测边界，不应作为生产默认常态。**

### 11.5 那“每租户 1 分区”到底对不对？

答案是：对很多 SaaS 场景来说，这是一条合理起点，但要带约束条件。

合理的地方：

1. 语义清晰，租户边界天然明确。
2. 排障方便，租户级 backlog 和延迟好定位。
3. 配合细粒度 ACK，慢租户可以局部堆积不拖全局。

要注意的地方：

1. “每租户 1 分区”本质是把租户规模直接映射到分区规模。
2. 随租户增长，客户端和运维面状态都会增长。
3. 大租户仍然需要独立策略，不能长期跟长尾租户同池竞争。

### 11.6 我们建议的落地方式：先分桶，再扩副本

相比“一个正则吃全量”，我们更推荐“正则分桶 + 每桶扩副本”：

1. 先按租户命名规则做桶，例如 16/32/64 个桶。
2. 每个消费者组只订阅一个桶的 topic 集合。
3. 每桶内部用 shared subscription 进行副本扩缩容。
4. 大租户从桶里剥离到专用消费者池。

这个顺序很关键，因为“全量正则 + 加副本”不一定降低单实例状态压力。  
先把单实例承载范围切小，再在桶内横向扩，整体会稳定得多。

这一步也经常被质疑“是不是架构倒退”。  
我的看法是：如果你的团队并不以调度系统为主产品，那么静态或半静态分桶不是倒退，而是有意识地降复杂度。  
它的代价是灵活性下降，收益是系统行为更可预测、值班更可控。

### 11.7 一套可执行的经验阈值（用于起步，不是金科玉律）

在缺少历史数据时，我们会先给一个偏保守的初始约束：

1. 单实例承载 topic（或 topic-partition）控制在 200-500 起步。
2. `receiverQueueSize` 从小值开始，随压测逐步放大，不反过来。
3. 自动扩容触发指标不只看 CPU，还要看 backlog、处理延迟、unacked、GC。
4. 大租户定义阈值后单独拆池，避免长尾租户被波峰绑架。

这组阈值不是要你照抄，而是避免“上线第一天就把故障域做太大”。

### 11.8 Kafka 对比：不是“Kafka 不行”，而是“匹配度不同”

这点必须说清楚，避免制造稻草人。  
在“EPS 3000、Topic 几千”的规模上，Kafka 完全有机会跑起来，也有大量业界实践。

我们选择 Pulsar，不是因为 Kafka 在这个规模必败，而是基于以下匹配差异：

1. 我们更需要 tenant/namespace 级语义与权限边界。
2. 我们更依赖 per-message ACK/NACK 语义来简化进度治理。
3. 我们希望把长尾租户自动发现与分层治理结合到同一套模型里。

如果你的团队已经把 Kafka 的多租户治理体系打磨得很好，继续用 Kafka 也完全合理。  
选型不该是“神话对神话”，而应是“问题空间对能力边界”。

## 十二、这次复盘真正成立的结论是什么

回头看这次重构，真正成立的不是“我们找到了银弹”，而是下面这些更朴素的工程事实：

1. ACK 只能解决进度，不解决资源隔离，后者必须单独设计。
2. 正则订阅能降接入成本，但会引入元数据和重建开销，需分层使用。
3. 分桶仍然是调度，只是弱中心调度，更适合非调度型团队的主战场。
4. Kafka 与 Pulsar 都能做事，关键是你的约束与团队能力匹配哪一边。

一句话总结这次演进：  
**我们不是从一个“正确技术”迁移到另一个“更正确技术”，而是把系统从“靠经验硬顶”改成“能力边界清晰、可验证、可运维”的状态。**

这背后最大的变化，是团队愿意承认取舍，不把妥协包装成胜利，不把工具包装成信仰。  
对多租户系统来说，这种诚实，往往比“某个漂亮架构图”更值钱。
