Repository Guidelines（仓库指南）

项目结构与模块组织

本仓库是一个基于 Jekyll 的 GitHub Pages 博客。

  • _posts/：博客文章，文件名格式为 YYYY-MM-DD-slug.md。
  • _layouts/：页面布局模板（如 default.html、post.html）。
  • _includes/：可复用的 Liquid 片段（如文章列表渲染）。
  • assets/css/、assets/js/：主题样式与前端脚本。
  • .github/workflows/pages.yml：构建并部署到 gh-pages 的工作流。
  • scripts/new-post.ps1：PowerShell 发文脚手架。
  • 根页面：index.md、archive.md、tags.md、about.md、404.md。

构建、测试与本地开发命令

在仓库根目录执行：

bundle install
bundle exec jekyll serve
bundle exec jekyll build
  • jekyll serve：本地预览（http://127.0.0.1:4000）。
  • jekyll build：生成生产构建到 _site/。

创建新文章（PowerShell）：

.\scripts\new-post.ps1 -Title "我的文章" -Description "摘要" -Tags 架构设计,SaaS化解读

编码风格与命名规范

  • Markdown、HTML、CSS、脚本统一使用 UTF-8。
  • 保持与现有文件一致的缩进风格（HTML/CSS/Liquid 使用 2 空格）。
  • 面向站点展示的文案优先中文，表达简洁直接。
  • 文章 frontmatter 固定使用：title、description、date、updated、tags。

决策优先级与冲突处理

当规则冲突或目标冲突时，按以下顺序决策（从高到低）：

  1. 真实性与安全性：不得伪造数据、误导读者或引入高风险改动。
  2. 可发布性：页面可构建、可渲染、关键路径不报错。
  3. 内容可读性：信息结构清晰、结论可验证、读者可执行。
  4. 一致性：遵循既有信息架构、文案口径、视觉 token。
  5. 美观度：在不破坏前四项前提下优化视觉表现。

执行约束：

  • 不以“视觉好看”为理由破坏内容真实性。
  • 不以“快速上线”为理由跳过构建与发布门禁。
  • 不以“炫技动画”为理由损害性能与可读性。

博客写作 SOP（从选题到发布）

每篇技术文章遵循以下流程，不跳步：

  1. 选题定义：写清问题、目标读者、适用边界、预期收益。
  2. 提纲评审：形成 5-8 个一级要点，明确证据与示例位置。
  3. 初稿产出：先完整表达，不在第一版过度打磨语气。
  4. 技术校验：核查术语、链路、代码片段、结论与反例。
  5. 可读性精修：优化标题层级、段落节奏、摘要与结尾行动项。
  6. SEO 与结构检查：标题、描述、slug、tag、内部链接一致。
  7. 发布前门禁：通过本文件“发布门禁清单”后再推送。

验收口径（每篇必须满足）：

  • 开头 200 字内说清“问题是什么、为什么重要”。
  • 至少包含 1 处 trade-off 和 1 处不适用边界。
  • 至少包含 1 个可复用清单、步骤或决策框架。

Frontmatter 元数据规范（必填 + 可选）

必填字段：

  • title：文章标题。
  • description：摘要，建议 50-120 字。
  • date：首次发布时间（固定）。
  • updated：最近更新时间（可变）。
  • tags：标签数组，2-6 个为宜。

可选字段（用于首页运营与展示）：

  • cover：封面图路径或 URL。
  • hero_style：封面风格标识（如 minimal, technical）。
  • series：专题名（如 国际化治理）。
  • series_part：专题内序号（从 1 开始递增）。
  • series_intent：专题目标（一句话说明该专题解决什么问题）。
  • series_buildable：是否具备专栏化价值（true/false）。
  • featured：是否精选（true/false）。
  • featured_rank：精选排序权重，值越小优先级越高。
  • hot：是否热门（true/false）。
  • hot_rank：热门排序权重，值越小优先级越高。
  • draft：草稿状态（true 表示不应上线）。
  • canonical_url：规范化链接（转载或多站同步时使用）。

字段约束：

  • featured=true 或 hot=true 时，必须填写 description。
  • draft=true 的文章不得进入首页、热门、专题模块。
  • 同一文章可同时属于 featured 与 hot，但需设置不同 rank。
  • series_buildable=true 时，必须填写 series、series_part、series_intent。
  • series_part 必须为正整数，且同一 series 内应保持递增。

自动标签与自动运营字段规则（写作时自动完成）

目标：作者在新建文章时无需手工填写标签与运营字段。

自动标签规则（按标题 + 摘要关键词匹配）：

  • 国际化：国际化、i18n、locale、多语言、时区、UTC。
  • 架构设计：架构、DDD、领域、状态机、适配器、边界。
  • 工程治理：治理、规范、门禁、checklist、流程、标准化、自动化。
  • SRE：故障、告警、监控、可用性、稳定性、恢复、容量。
  • 性能优化：性能、延迟、吞吐、CPU、内存、IO、GC、压测。
  • Linux：Linux、top、vmstat、pidstat、iostat、内核、系统调用。
  • ToB：ToB、B端、企业客户、SaaS。

自动专题（series）规则：

  • 命中 国际化 => series: "全球化工程治理"。
  • 命中 SRE 或 Linux => series: "SRE 故障实战"。
  • 命中 架构设计 或 工程治理 => series: "工程治理体系"。

自动专栏可构建性评估规则：

  • 每篇文章创建时必须自动评估 series_buildable，不允许跳过。
  • 命中 series 自动映射规则时，默认 series_buildable: true。
  • series_buildable=true 时自动补齐：
    • series_part：按同系列历史文章数 + 1 生成。
    • series_intent：按系列模板自动生成，作者可在发文时微调。
  • 未命中系列映射时，默认 series_buildable: false，后续可手动提升为专栏。

自动热门（hot）规则：

  • 命中 架构设计 / 工程治理 / SRE / 性能优化 任一主题信号时，默认 hot: true。
  • 若仅命中兜底标签且缺乏主题信号，默认 hot: false。
  • hot_rank 默认按内容价值粗排（高优先主题给更小 rank），允许后续手动微调。

自动相关文章规则：

  • 文章页优先按 series 推荐相关文章。
  • 若同系列不足，回退按标签重叠度补足。
  • 默认展示 3 篇，且排除当前文章与 draft=true 文章。

关联文章质量规则（发布门禁）：

  • 每篇文章发布前必须满足“可推荐数量 >= 3”（同系列 + 标签回退）。
  • 推荐结果需覆盖“至少 1 篇基础向 + 1 篇进阶向”。
  • 推荐列表禁止包含当前文章、草稿文章、失效链接。

首页运营规则（元数据驱动）

首页模块必须由 frontmatter 驱动，不允许硬编码文章 URL。

模块规则：

  • 精选文章：featured=true，按 featured_rank 升序，再按 updated 降序。
  • 热门文章：hot=true，按 hot_rank 升序，再按 updated 降序。
  • 专题连载：按 series 字段聚合；若缺失可回退到约定标签映射。
  • 状态条：仅展示真实可核查指标（总文章数、最近更新时间、专题数量）。

运营原则：

  • 首页不展示“精确访问量”作为主要卖点。
  • 若需“热度感”，优先使用“热门/精选/连载”标签体系。
  • 每次发新文后，检查是否需要更新 featured/hot/series。

发布门禁清单（发布前必须通过）

内容与渲染门禁：

  1. bundle exec jekyll build 构建通过。
  2. 关键页面渲染正常：/、/archive/、/tags/、/about/。
  3. 新文 permalink、标签锚点、目录锚点可跳转。
  4. 首页精选/热门/专题卡片无空字段、无死链接。

质量门禁：

  1. Frontmatter 字段完整且格式合法。
  2. 无明显硬编码运营数据（如虚构访问量、伪造统计）。
  3. 文章符合“技术博客质量标准”与写作 SOP。
  4. 专栏评估字段合法（series_buildable/series_part/series_intent 一致）。
  5. 相关文章推荐满足最小数量与质量规则。
  6. 推送后 GitHub Actions 的 Pages 工作流成功。

视觉系统约束（Design Tokens 与改版边界）

允许优先调整：

  • 颜色 token：--text、--muted、--accent、--line。
  • 节奏 token：字号层级、行高、间距、卡片内边距。
  • 交互强度：hover、阴影、边框对比、动效时长。

谨慎调整（需明确理由）：

  • 布局结构（模块顺序、信息架构、导航结构）。
  • 文章页阅读路径（封面、目录、正文宽度核心逻辑）。
  • 标签与归档页面的数据组织方式。

禁止事项：

  • 为追求“炫酷”引入影响阅读的重动效与噪点背景。
  • 无依据地切换整体视觉风格，导致站点品牌漂移。
  • 在未定义 token 的情况下散落硬编码色值与尺寸。

技术博客质量标准（写作沉淀）

目标：文章应帮助读者做出更好的技术决策，而不仅是“展示作者懂技术”。

七条质量标准

  1. 问题定义清楚：交代业务场景、规模、约束、成功标准。
  2. 结论可证伪：给出判断依据与选择条件，说明为何选 A 不选 B。
  3. 过程可复现：提供关键输入、架构边界、核心实现与落地路径。
  4. 权衡讲透：明确收益、成本、风险、维护负担与失败模式。
  5. 边界明确：写清适用场景与不适用场景，避免“万能方案”叙事。
  6. 结果可验证：尽量给出前后对比指标，至少给观察指标和验证方法。
  7. 结构服务决策：优先采用“背景 -> 决策点 -> 方案 -> 权衡 -> 落地 -> 结果 -> 复盘”。

四个低质量信号

  1. 术语密集但缺乏上下文与约束。
  2. 只有成功案例，没有踩坑、回滚和失败复盘。
  3. 只有代码片段，没有系统边界与链路影响说明。
  4. 只有“最佳实践”结论，没有“为什么是这套实践”的推导过程。

发文前自检清单

  1. 是否在开头 200 字内明确“问题是什么、为什么重要”。
  2. 是否明确写出 trade-off，而不是只写优点。
  3. 是否包含至少一个反例或不适用边界。
  4. 是否给出可复用的检查清单、落地步骤或迁移顺序。
  5. 是否提供验证方式（指标、日志、SLO、回滚点或验收口径）。
  6. 是否避免空话和口号式表达，段落能对应具体工程动作。

测试与验证指南

本仓库无单元测试框架，采用构建与页面核验：

  1. 本地执行 bundle exec jekyll build，确保构建通过。
  2. 抽查关键页面：/、/archive/、/tags/、/about/。
  3. 推送后确认 GitHub Actions 的 Deploy blog to GitHub Pages 任务成功。

提交与 Pull Request 规范

  • 提交信息遵循 Conventional Commits，例如：
    • feat(theme): 重构为暗色阅读优先主题
    • fix(ci): ...
    • content(profile): 更新关于页与标签体系
  • 每次提交聚焦单一主题（主题样式、内容、CI、脚本分开提交）。
  • PR 建议包含：
    • 变更内容与动机。
    • 受影响的页面或文件路径。
    • 涉及 UI 时附截图。
    • 本地构建或 CI 通过说明。

Git 自动提交策略

  • 完成代码修改后，默认直接执行提交，不再二次询问。
  • 提交格式使用 Conventional Commits。
  • 允许执行：git add、git commit。
  • 禁止执行：git reset --hard、git checkout --、git revert（除非用户明确要求）。
  • 若工作区存在与当前任务无关的改动，不要回滚，跳过并只提交本次相关文件。
