# 博客大纲 Doctor

## 核心职责

这个 skill 只管四件事：

1. 开头有没有把背景、前提、当前讨论边界交代清楚。
2. 文章框架是不是一条线往前推，而不是写到哪想到哪。
3. 二级、三级标题之间有没有真实关系，而不是东拼西凑。
4. 开头承诺和后文展开是否对得上。

默认不处理这些事：

1. 全文润色。
2. 文风美化。
3. SEO、运营字段、发布门禁。
4. 技术正确性以外的页面样式问题。

## 默认任务理解

1. 目标不是“生成一篇标准博客”，而是把作者真实想法整理成可发布的大纲和结构。
2. 先判断问题是出在标题、结构、叙事主线，还是表达抽象度，不要看到标题别扭就只修标题。
3. 如果标题之间像拼起来的，优先判断是主线没立住，还是标题本身措辞差；通常先修主线，再修标题。
4. 默认职责是整理、压缩、提纯作者已经表达过的结构判断，不是替作者脑补新论点，也不是替作者临时发明一套看起来工整的框架。
5. 如果素材不足以支撑某个大纲判断，优先指出缺什么，不要自动补成一套表面完整、实际空心的结构。
6. 技术文章默认先从业务触发点起手，再进入工程约束和技术处理，不要从抽象技术主题直接起笔。
7. 如果文章讨论的是性能、治理、架构、模型这类大词，先检查业务为什么会在这个阶段把这个问题逼出来，而不是先讲概念。
8. 如果文章表面在讲业务矛盾，正文却越来越像在单独讲某个机制，优先判断为主抓手选错，而不是局部段落没修顺。
9. 如果某个机制、事故或触发事件只是把问题逼出来的入口，不要直接把它写成全文题眼；题眼应该落在文章真正要回答的对象上，比如这类需求怎么接、怎么推进、怎么收口。
10. 如果一个核心痛点一两段就能讲清，不要为了结构对称把它拆成多个二级标题，靠反复换抽象词或重新命名同一个意思来硬撑篇幅。
11. 不要默认把二级标题写成 `一、二、三、四` 这种编号骨架。只有当文章本来就是稳定的并列清单、步骤流程或检查项时，编号才有价值；如果正文是在连续推进判断，强行编号只会制造假结构感。
12. 不要默认把“结论先行”当成开头模板。开头要立住的是问题、边界和阅读抓手，不一定非要先把最后答案端出来。
13. 标题不要写成作者的自我旁白或写作动作，例如 `这篇最后该留下什么`、`后来我最后怎么定下来的`。标题优先回答“这一节在讲什么”，而不是“作者准备怎么总结”。
14. 如果一篇文章里同时长出了多个独立主判断，默认先判断它是不是专栏总起或系列入口；不要先假设它一定能靠重排大纲收成一篇。
15. 如果文章的真实张力来自“旧系统已经被证明有价值，新系统是从旧系统里分出来的，但新系统要单独申请资源”，开头必须先把这层承接关系写明。不要一上来把它抽象成泛泛的“新业务要证明价值”，否则资源矛盾会失真。
16. 如果文章明确承接上一篇，开头必须先交代上一篇已经回答了什么、这一篇只继续补哪一段，不要只写一句“上一篇聊过”。否则续篇很容易和前文重合，读起来像同一篇拆两半。

## 使用方式

遇到下面这些情况时使用这个 skill：

1. 用户说“大纲不顺”“标题像拼起来的”“二级标题没关系”“三级标题乱”。
2. 文章开头已经承诺了几个问题、几种方案、几条线，但后文展开对不上。
3. 正文里长出了分片、Router、上游改模型这类关键节点，但开头没交代。
4. 文章读起来像“有框架在指引”，而不是事情自然往下走。
5. 用户明确说“这篇应该拆成多个主题”“这更像一个专栏”“一篇里抛了太多东西”。

## 检查顺序

### 1. 先看开头有没有立住

开头至少要说清楚这几件事：

1. 当时系统处在什么阶段。
2. 哪些前提已经定了，不再讨论了。
3. 现在只讨论什么问题。
4. 真正卡住什么。
5. 为什么这个问题值得读下去。

如果这是技术文章，再补一条默认检查：

1. 业务现场有没有出现。
2. 是哪类业务、哪类结果、哪类处理差异把问题逼出来的。
3. 读者能不能看懂作者为什么会在这个点开始重新看这个问题。
4. 如果背景涉及本地化交付转 SaaS、项目制转产品化、私有化转共享化等阶段切换，开头是否把“旧形态是什么”“当前最优先验证什么”说出来。

如果这四件事没有立住，后面的大纲再完整也容易发飘。

再补一个反误判：

1. “开头成立”不等于“前两段必须先给最终结论”。
2. 如果文章的价值在于把答案推出，例如过程文、观察文、部分判断文，默认允许先立场景、分叉和问题，再在后文把答案走出来。
3. 开头不要先解释作者准备怎么写，例如“这篇先讲背景”“先把前提摆一下”。如果这句话不承载业务信息，默认删掉。

另外再补一条：

1. 如果这是独立文章，开头必须能独立成立，不能默认读者看过上一篇。
2. 如果这是承接上文的文章，就直接写清楚承接关系，不要写成既没承接、又没背景的状态。

### 2. 再看文章承诺和正文展开是不是一致

如果开头写了“这篇只讲三种做法 / 三个问题 / 三条线”，后文就必须按这个承诺展开。

如果后面自然长出了新节点，要做二选一：

1. 把它收回原有结构里，作为某一节的自然展开。
2. 回头重写开头，把这个新节点提前交代进去。

不允许出现这种情况：

1. 开头说三种做法。
2. 正文写着写着突然飞出一个新方案。
3. 结尾又多出一个开头没提过的方向。

### 3. 再看二级标题是不是在往前推

先补一条总规则：

1. 一级标题要围绕同一条推进线展开，优先写对象、动作、边界、分层、代价，不要写成一组观点卡片。
2. 一级标题通常控制在 5 到 7 个；如果更多，先检查是不是把并列信息误写成主线节点。
3. 当文章更适合“建模拆解”或“承载分层”时，标题应围绕系统对象推进，而不是围绕作者感受推进。
4. 不要为了把标题写得“有判断”而写成一句端着的评论话。标题首先要像正常人在往下讲事情，其次才是提炼观点。
5. 默认先走“业务问题 -> 工程约束 -> 技术处理 -> 代价与边界”这条线，不要先写成“技术概念 -> 能力清单 -> 总结判断”。
6. 标题优先靠语义推进，不要靠编号制造秩序感。编号只能标记已经存在的结构，不能替代结构本身。

二级标题必须承担推进职责。检查时至少过这四条：

1. 这一节和上一节之间有没有清楚的因果或推进关系。
2. 标题能不能直接落到正文第一段，而不是标题很大、正文很虚。
3. 去掉这个标题后，整篇会不会更顺。如果会，说明它大概率是硬塞进去的。
4. 这一节是不是只是在重复前文，只是换了一个更像总结的说法。
5. 标题首先要让读者知道这一节在回答什么问题，解决什么卡点，或者把前文往哪里推进，而不是只追求“像结论”或“像点评”。
6. 这一节是不是在回应一个真实业务或工程卡点，而不是为了讲某个机制单独起了一节。
7. 这一节是不是把某个支撑机制写成了主角，结果把全文真正的问题带跑了。
8. 把标题前面的编号去掉以后，这条推进线还成不成立；如果去掉编号后立刻显得散，说明原来靠的是格式，不是结构。
9. 如果这一节在引入一个新系统、新机制或新角色，正文第一段是否把它最小工作模型说出来；如果连“它大概怎么工作”都没交代，读者通常跟不上为什么会讨论它。
10. 如果把其中任何一个二级标题单独拿出来，都能独立发展成一篇新文章，先判断是不是系列拆分问题，而不是继续硬做单篇结构。

总原则只有一条：  
标题必须有逻辑。只要二级标题之间没有清楚关系，读起来像拼出来的，大纲就算失败。

二级标题常见问题：

1. 只是在评论，不是在推进。
2. 写成一句端着的判断话。
3. 标题像观点卡片，读不出顺序。
4. 标题之间没有“上一节为什么走到下一节”的关系。
5. 同一个意思被拆成两到四节反复讲，只是换了叫法，例如从“系统行为变了”换成“举例验证变多了”“按版本验证失效了”，但正文没有新增对象、状态或代价。

### 4. 最后看三级标题是不是服务对应二级标题

三级标题只能做两件事：

1. 把一个二级问题拆成几个连续步骤。
2. 把一个二级问题拆成几个并列观察点。

三级标题检查时至少过这三条：

1. 是不是同一个二级问题下的自然拆分，而不是另起一题。
2. 拆分维度是不是一致，不能一个讲风险、一个讲方案、一个讲感受。
3. 子标题读下来能不能看出顺序，而不是几句并列判断堆在一起。

## 重排原则

当大纲有拼凑感时，优先按这个顺序处理：

1. 先重写开头，立住背景和讨论边界。
2. 再重排二级标题，让它们形成一条推进线。
3. 再处理三级标题，让它们服从二级标题。
4. 最后才改标题措辞。
5. 如果前四步之后仍然像“多个主题并排摆放”，停止单篇重排，改做系列拆分方案。

不要反过来做。  
如果框架是散的，只改标题用词，读起来还是会像拼出来的。

再补一条主线修正原则：

1. 如果文章真正想回答的是业务矛盾，就不要让 batch、DLQ、ack、Router 这类中间机制成为主轴。
2. 机制只能作为支撑点，解释它怎样接住前文已经立住的业务卡点。
3. 一旦发现正文越来越像“专门解释某个机制”，优先回头重定题眼，而不是继续补机制细节。

## 输出要求

使用这个 skill 时，默认输出两部分：

1. 当前大纲的问题：
   - 哪些标题断线了。
   - 哪些标题是硬塞的。
   - 哪些关键节点开头没承诺。
   - 哪些地方没有先立业务触发点。
   - 哪些地方在先讲技术主题，后补业务落点。
   - 哪些地方已经被某个机制带跑，偏离了文章原本要回答的问题。
2. 重排后的大纲：
   - 先给新的二级标题线。
   - 再补必要的三级标题。
   - 每个标题只保留推进职责，不写点评腔。
   - 如果更适合拆系列，直接给出“总题眼 + 拆分后的 3 到 6 篇候选标题”，不要硬交一版假完整单篇大纲。

## 红线

1. 不要只修标题措辞，不修结构。
2. 不要一边写正文一边临时长标题。
3. 不要把正文里临时想到的新判断直接升成一个新标题。
4. 不要让标题看起来像“模型在做总结”，而不是“人在往下讲事情”。
5. 不要先写抽象技术主题，再回头补业务理由。
6. 不要为了讲某个机制、能力或概念，硬开一节和主线脱开的标题。
7. 不要明明在写业务问题，最后却把文章写成某个机制的专题说明。
8. 不要在结尾突然切到抽象收束、专题导流或“相关文章推荐腔”，让正文像突然换了频道。
9. 不要因为某个触发器很抓人，就让标题和一级标题都围着它打转；如果它只是引子，正文必须尽快回到真正的主问题。
10. 不要为了显得工整，给本来不是清单的文章硬套 `1/2/3/4/5` 或 `一/二/三/四/五`。这种格式很容易把判断文写成提纲展开稿。
