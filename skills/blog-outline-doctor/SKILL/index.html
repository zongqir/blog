博客大纲 Doctor

核心职责

这个 skill 只管四件事：

  1. 开头有没有把背景、前提、当前讨论边界交代清楚。
  2. 文章框架是不是一条线往前推，而不是写到哪想到哪。
  3. 二级、三级标题之间有没有真实关系，而不是东拼西凑。
  4. 开头承诺和后文展开是否对得上。

默认不处理这些事：

  1. 全文润色。
  2. 文风美化。
  3. SEO、运营字段、发布门禁。
  4. 技术正确性以外的页面样式问题。

默认任务理解

  1. 目标不是“生成一篇标准博客”，而是把作者真实想法整理成可发布的大纲和结构。
  2. 先判断问题是出在标题、结构、叙事主线，还是表达抽象度，不要看到标题别扭就只修标题。
  3. 如果标题之间像拼起来的，优先判断是主线没立住，还是标题本身措辞差；通常先修主线，再修标题。
  4. 默认职责是整理、压缩、提纯作者已经表达过的结构判断，不是替作者脑补新论点，也不是替作者临时发明一套看起来工整的框架。
  5. 如果素材不足以支撑某个大纲判断，优先指出缺什么，不要自动补成一套表面完整、实际空心的结构。

使用方式

遇到下面这些情况时使用这个 skill：

  1. 用户说“大纲不顺”“标题像拼起来的”“二级标题没关系”“三级标题乱”。
  2. 文章开头已经承诺了几个问题、几种方案、几条线，但后文展开对不上。
  3. 正文里长出了分片、Router、上游改模型这类关键节点，但开头没交代。
  4. 文章读起来像“有框架在指引”，而不是事情自然往下走。

检查顺序

1. 先看开头有没有立住

开头至少要说清楚这几件事：

  1. 当时系统处在什么阶段。
  2. 哪些前提已经定了，不再讨论了。
  3. 现在只讨论什么问题。
  4. 真正卡住什么。

如果这四件事没有立住，后面的大纲再完整也容易发飘。

另外再补一条：

  1. 如果这是独立文章，开头必须能独立成立，不能默认读者看过上一篇。
  2. 如果这是承接上文的文章，就直接写清楚承接关系，不要写成既没承接、又没背景的状态。

2. 再看文章承诺和正文展开是不是一致

如果开头写了“这篇只讲三种做法 / 三个问题 / 三条线”，后文就必须按这个承诺展开。

如果后面自然长出了新节点，要做二选一：

  1. 把它收回原有结构里，作为某一节的自然展开。
  2. 回头重写开头，把这个新节点提前交代进去。

不允许出现这种情况：

  1. 开头说三种做法。
  2. 正文写着写着突然飞出一个新方案。
  3. 结尾又多出一个开头没提过的方向。

3. 再看二级标题是不是在往前推

先补一条总规则：

  1. 一级标题要围绕同一条推进线展开，优先写对象、动作、边界、分层、代价，不要写成一组观点卡片。
  2. 一级标题通常控制在 5 到 7 个；如果更多，先检查是不是把并列信息误写成主线节点。
  3. 当文章更适合“建模拆解”或“承载分层”时，标题应围绕系统对象推进，而不是围绕作者感受推进。
  4. 不要为了把标题写得“有判断”而写成一句端着的评论话。标题首先要像正常人在往下讲事情，其次才是提炼观点。

二级标题必须承担推进职责。检查时至少过这四条：

  1. 这一节和上一节之间有没有清楚的因果或推进关系。
  2. 标题能不能直接落到正文第一段，而不是标题很大、正文很虚。
  3. 去掉这个标题后，整篇会不会更顺。如果会，说明它大概率是硬塞进去的。
  4. 这一节是不是只是在重复前文，只是换了一个更像总结的说法。
  5. 标题首先要让读者知道这一节在回答什么问题，解决什么卡点，或者把前文往哪里推进，而不是只追求“像结论”或“像点评”。

总原则只有一条：
标题必须有逻辑。只要二级标题之间没有清楚关系，读起来像拼出来的，大纲就算失败。

二级标题常见问题：

  1. 只是在评论，不是在推进。
  2. 写成一句端着的判断话。
  3. 标题像观点卡片，读不出顺序。
  4. 标题之间没有“上一节为什么走到下一节”的关系。

4. 最后看三级标题是不是服务对应二级标题

三级标题只能做两件事：

  1. 把一个二级问题拆成几个连续步骤。
  2. 把一个二级问题拆成几个并列观察点。

三级标题检查时至少过这三条：

  1. 是不是同一个二级问题下的自然拆分，而不是另起一题。
  2. 拆分维度是不是一致，不能一个讲风险、一个讲方案、一个讲感受。
  3. 子标题读下来能不能看出顺序，而不是几句并列判断堆在一起。

重排原则

当大纲有拼凑感时，优先按这个顺序处理：

  1. 先重写开头，立住背景和讨论边界。
  2. 再重排二级标题，让它们形成一条推进线。
  3. 再处理三级标题，让它们服从二级标题。
  4. 最后才改标题措辞。

不要反过来做。
如果框架是散的，只改标题用词，读起来还是会像拼出来的。

输出要求

使用这个 skill 时，默认输出两部分：

  1. 当前大纲的问题：
    • 哪些标题断线了。
    • 哪些标题是硬塞的。
    • 哪些关键节点开头没承诺。
  2. 重排后的大纲：
    • 先给新的二级标题线。
    • 再补必要的三级标题。
    • 每个标题只保留推进职责，不写点评腔。

红线

  1. 不要只修标题措辞，不修结构。
  2. 不要一边写正文一边临时长标题。
  3. 不要把正文里临时想到的新判断直接升成一个新标题。
  4. 不要让标题看起来像“模型在做总结”，而不是“人在往下讲事情”。

持续记录那些真正影响结果的工程取舍。
