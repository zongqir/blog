文章事实风险审查

核心定位

这个 skill 不负责做“绝对真伪判定”。

它负责做的是：

  1. 找出文章里哪些内容可能是假的。
  2. 找出哪些内容不像假的，但证据不够。
  3. 找出哪些内容其实只是推测，却被写成了事实。
  4. 找出哪些技术结论、时间表述、组织关系、规模数字容易出错。

也就是说，它输出的是事实风险，不是“最终判决”。

重点检查什么

1. 伪造经历和伪造案例

优先抓这些句子：

  1. 我当时做了……
  2. 我们线上已经……
  3. 后来验证下来……
  4. 实际结果证明……

如果作者没有明确给过这类经历素材，这些句子都属于高风险。

2. 把推测写成事实

优先抓这些情况：

  1. 没有证据，却写成确定结论。
  2. 本来只是可能原因，却写成唯一原因。
  3. 本来只是行业常见情况，却写成“我们就是这样”。

重点看这些词：

  1. 一定
  2. 就是
  3. 本质上就是
  4. 真实情况是
  5. 最后证明

3. 技术结论超出证据

典型问题：

  1. 从一个局部现象推到系统级结论。
  2. 只写一个大方向判断，没有证据位。
  3. 用正确但空的技术话替代真实风险。

比如：

  1. 只写“内存压力很大”，不写连接、队列、ack、cursor、backlog、恢复过程。
  2. 只写“隔离性变差”，不写到底谁影响谁。
  3. 只写“稳定性下降”，不写重启、恢复、故障域。

4. 时间、版本、规模、组织关系

这些地方最容易不小心写错：

  1. 时间线：当时、后来、现在 到底分别指什么。
  2. 版本和阶段：系统在哪个阶段，哪些东西已经上线，哪些还只是讨论。
  3. 规模数字：topic 数、租户数、partition 数、实例数，是事实还是举例。
  4. 组织关系：是不是独立部门、是不是跨团队协作、是谁在推进这件事。

5. 引用上文和承接关系

如果文章承接上一篇，要检查：

  1. 上下文是否真的在前文出现过。
  2. 这篇是否仍然能让没看过上文的人知道最关键的前提。

如果文章是独立文章，就要检查有没有偷偷默认读者知道上文。

审查顺序

1. 先标“确定事实”和“作者判断”

每一段先分成两类：

  1. 确定事实：
    • 已发生的系统状态
    • 明确给出的数字
    • 明确给出的历史决策
  2. 作者判断：
    • 为什么这样想
    • 倾向哪种方案
    • 觉得哪里风险更大

这一步的目的，是避免把“判断”误写成“事实”。

2. 再找高风险句子

高风险句子通常有三类：

  1. 作者没提供素材却写得像亲身经历。
  2. 没有证据却写得像铁结论。
  3. 时间、规模、责任归属写得过满。

3. 再标证据等级

默认用这三档：

  1. 低风险：
    • 作者明确提供过素材。
    • 文章里有足够上下文支撑。
  2. 中风险：
    • 判断方向合理，但证据不够。
    • 建议改成“倾向于”“可能”“常见情况”。
  3. 高风险：
    • 像脑补经历。
    • 像伪造结果。
    • 像把推测写成事实。

输出要求

使用这个 skill 时，默认输出三部分：

  1. 事实风险清单：
    • 哪些句子风险最高。
    • 风险属于哪一类。
  2. 风险原因：
    • 为什么这句证据不够。
    • 为什么这句像脑补。
    • 为什么这句时间或规模可能写错。
  3. 改写建议：
    • 删掉
    • 改成推测
    • 改成作者判断
    • 补充事实前提或证据

红线

  1. 不要因为句子写得自信，就默认它是真的。
  2. 不要因为方向合理，就默认作者真的经历过。
  3. 不要把“可能是假的”直接写成“就是假的”。
  4. 不要只说“建议核实”，要指出为什么这里需要核实。

持续记录那些真正影响结果的工程取舍。
