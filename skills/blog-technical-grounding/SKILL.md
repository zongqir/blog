# 博客技术抓手补强

## 核心定位

这个 skill 处理的不是“写得顺不顺”，也不是“结构对不对”。

它处理的是另一类常见问题：

文章的方向没错，判断也可能是对的，但整篇读下来只有道理，没有技术本身。

对程序员作者来说，这类问题很伤文。  
不是因为每篇文章都必须写成教程，而是因为如果全文都停在原则、判断、取舍，读者很容易觉得：

1. 你在讲一套正确的话。
2. 但你没有真的把系统抓在手里。
3. 文章里缺少能落到工程对象上的东西。

这个 skill 的目标，就是给文章补上这些“抓手”。

它不追求把每篇文章都补成实现文。  
它追求的是：哪怕只补 1 到 3 个技术细节，也要让读者感觉这篇文章在讲系统，而不是在讲道理。

## 它解决什么问题

适合处理这些情况：

1. 文章一直在讲原则、取舍、判断，但没有讲对象、状态、动作。
2. 读者能听懂结论，却不知道这套判断在系统里到底落在哪里。
3. 方案文看起来像方案，其实只剩一层抽象框架。
4. 文章说了“该怎么做”，却没有最小工作流、最小机制、最小例子。
5. 文章写的是工程问题，但程序员读完抓不到任何技术细节。
6. 作者明确反馈“全篇都在讲大道理”“没有讲技术本身”“需要加一点方案细节”“得有点东西”。
7. 技术名词不少，但都停在名词层，没有转成具体抓手。

## 默认边界

这个 skill 默认不处理这些问题：

1. 文章总结构怎么重排。
2. 句子是否自然、是否去 AI 味。
3. frontmatter、SEO、标签、聚合位。
4. 需要严谨外部证据的事实核查。

如果问题主要是结构，优先用 `$blog-outline-doctor`。  
如果问题主要是内容太薄，优先用 `$blog-depth-reviewer`。  
如果问题主要是文风生硬，优先用 `$blog-writing-style`。  
如果问题主要是“讲的都是对的，但没有技术抓手”，优先用这个 skill。

补充边界：

1. 不要因为作者是程序员，就强行把每篇文章都写成教程。
2. 纯观察文、纯过程文、纯职业判断文，如果技术细节不是文章主价值，不要硬塞一堆实现。
3. 这个 skill 的任务不是“加很多细节”，而是“加对细节”。

## 强触发

出现这些信号时，默认触发本 skill：

1. “有点空”“太虚”“全是大道理”“没有讲技术本身”
2. “多加一点技术细节”“多一点技术方案细节”“多一点实战抓手”
3. “得有点东西”“程序员读者会想看更具体的”
4. “这篇理论多于技术细节”
5. “这个方案到底具体怎么做”

## 什么算技术抓手

不是只有代码才算技术细节。

下面这些东西都可以是有效的技术抓手：

1. 系统对象：
   例如 topic、subscription、bucket、worker、router、控制面、cursor、lease。
2. 运行机制：
   例如谁 watch、谁感知变化、谁更新分配、谁执行 handoff。
3. 最小工作流：
   例如新租户接入怎么发生，扩容怎么发生，故障接管怎么发生。
4. 规则和阈值：
   例如什么时候迁桶，什么时候拆大租户，哪些信号会触发调整。
5. 例子：
   例如 200 个租户怎么切 8 个桶，某个大租户怎么从共享桶里剥出来。
6. 故障面：
   例如一个实例挂掉，到底会影响哪些状态、哪些租户、哪些消息。
7. 代价落点：
   例如状态会沉到哪里，哪一层会开始变重，值班会追什么。

## 如何选要补的细节

默认不要平均加。

优先补这几类最值钱的细节：

1. 最能支撑文章主判断成立的那个机制。
2. 最能让“判断”落回系统对象的那个动作。
3. 最能让程序员读者抓住方案轮廓的那个例子。

默认顺序：

1. 先补一个“对象关系”。
2. 再补一个“运行机制”。
3. 如果还需要，再补一个“例子或故障面”。

一篇文章哪怕只补到这三层里的两层，通常也会比全篇抽象判断更稳。

## 具体怎么补

### 1. 把抽象判断压回对象

如果原文写：

- “状态变重了”
- “运行时关系更复杂了”
- “系统更难治理了”

默认追问：

1. 哪个状态？
2. 落在谁手里？
3. 是谁更难看清？

### 2. 把对象补成动作

不要只报组件名。

如果原文里出现：

- topic
- consumer
- subscription
- router
- etcd

默认要回答一句：

它在这篇文章里到底干什么。

### 3. 给一个最小例子

例子不用大。

但最好至少有一种：

1. 数量例子：
   例如 200 个租户、8 个桶、某个实例挂了会波及多少。
2. 流程例子：
   例如新租户进来之后先写注册表，再改分配，再由 worker watch 到变化。
3. 故障例子：
   例如单实例重启后为什么之前是固定几个 topic 受影响，后来变成一批状态被重新分发。

### 4. 补“它为什么有用”

不是把技术细节堆上去就行。

每补一个细节，都要回答一句：

这个细节为什么对当前判断重要。

例如：

1. 它解释了为什么这条路好改。
2. 它解释了为什么故障域会变大。
3. 它解释了为什么系统后面会越来越重。

## 常见误区

### 1. 不要把“技术抓手”误做成“术语堆砌”

错法：

1. 连续报一串组件名。
2. 写很多实现词，但没有关系和动作。
3. 为了显得专业，往里塞不服务主线的机制。

### 2. 不要把文章补成教程

这个 skill 不是逼每篇文章写完整实现。

不要默认补这些东西：

1. 大段代码。
2. 完整配置文件。
3. 全链路伪代码。
4. 超出文章主线的实现枝节。

### 3. 不要伪造技术细节

1. 不确定的参数，不要编。
2. 不确定的阈值，不要编。
3. 不确定的组件行为，不要装成确定事实。

如果只能写到“最小合理轮廓”，就老实写轮廓。

## 默认输出

使用这个 skill 时，默认输出三部分：

1. 当前文章最缺的技术抓手：
   - 指出 1 到 3 个最关键的缺口。
2. 为什么这些抓手必须补：
   - 说明不补的话，文章为什么会像在讲大道理。
3. 直接补写后的版本：
   - 直接把技术抓手写进正文。
   - 不擅自把文章改成教程。

## 红线

1. 不要为了“有技术感”乱补术语。
2. 不要把文章从判断文硬改成实现文。
3. 不要用假参数、假配置、假例子撑技术感。
4. 不要为了技术细节，把文章主线打断到看不见。
