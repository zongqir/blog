博客文风

这个 skill 管什么

这个 skill 只管“怎么写”，不管“怎么搭大纲”。

它主要处理这些问题：

  1. 文章读起来像模型在总结，不像人在复盘。
  2. 语言太高级、太抽象、太像报告。
  3. 说教味、布道味、营销味、第二人称太重。
  4. 中文表达不自然，读起来像翻译腔、摘要腔或修得太圆。
  5. 背景没立住，正文像悬空分析。
  6. 真实判断被修成四平八稳的中性正确。
  7. 技术细节看起来对，但没有泥土感。
  8. 为了去 AI 味，把真正有价值的内容一起删掉了。
  9. 技术密度过高，术语、组件、机制堆得很满，但读者抓不住主线。

它默认不处理这些问题：

  1. 二级三级标题怎么排。
  2. 整体大纲怎么重构。
  3. 哪些段落本身应该继续展开、写透，而不是精简。
  4. 发布门禁和运营字段。

如果问题主要是大纲、框架、标题推进关系，优先使用 $blog-outline-doctor。 如果问题主要是内容太薄、关键判断没写透，优先使用 $blog-depth-reviewer。

  2. 其中和文风最相关的口径是：保留人味，不装人味。

写作红线

1. 真实性优先

  1. 不要伪造经历、案例、踩坑、数据、结果、对比。
  2. 作者没说过的经验，不要补成“我曾经如何如何”。
  3. 不确定的内容，要明确写成推测、常见情况、可能原因，不要写成确定事实。

2. 不要把文章写成标准答案

  1. 目标不是“生成一篇标准博客”，而是把作者真实想法整理清楚。
  2. 不要把作者写成老师、导师、布道者。
  3. 如果素材不够，先指出缺什么，不要硬补成一篇完整但空心的文章。
  4. 不是每篇都要有完整结论、统一方法论和标准收束。
  5. 如果问题本来还在演进，允许停在当前判断、当前边界、当前倾向。

3. 不要有 AI 味

默认优先清理这些东西：

  1. 模板化表达。
  2. 正确但没信息量的空话。
  3. 过于高级的抽象词。
  4. 元说明腔。
  5. 方法论广播腔。

少写这些句子：

  1. 随着……发展
  2. 在当今……背景下
  3. 本文将……
  4. 值得注意的是
  5. 综上所述
  6. 先把问题问对
  7. 如果我要判断
  8. 真正值得写的是

再补几类高风险信号：

  1. 句子和段落过度平均。
  2. 文章被补得过度完整。
  3. 看起来高级、其实谁都能说的假洞察。
  4. 组件名很多，但现场感很弱。

再补一类常见失衡：

  1. 技术密度过高，但密度没有转成可理解的判断。
  2. 一段同时塞进概念、实现、比较、延伸，读者读完只记得术语很多。

4. 不要有说教味和假熟味

  1. 不要写“你应该”“你必须”“正确做法是”“最佳实践是”“永远不要”。
  2. 不要写“你一定也遇到过”“朋友，如果你看到这里”“你是不是也有这种感觉”。
  3. 默认用第一人称或中性叙述，不用第二人称带读者走。
  4. 不要写成教科书、百科、培训材料。

5. 中文表达约束

目标不是“像中文”，而是更接近中文技术写作者的自然表达。

优先遵守这些约束：

  1. 避免英译中句式。
  2. 避免论文摘要腔、咨询报告腔、产品宣讲腔。
  3. 允许句子短一些、顿一些，不强行做英文式平滑连接。
  4. 一个概念第一次定了，后面可以重复叫同一个名字，不必为了“文采”反复换说法。
  5. 优先保留中文技术写作里常见的推进方式：先交代背景，再说当时怎么想，再说后来卡在哪。

少用这类翻译腔句式：

  1. 本质上是对于……的重新思考
  2. 其核心在于……
  3. 围绕……展开了深入讨论
  4. 从某种意义上来说

中文技术文章更适合：

  1. 交代约束。
  2. 说明动作。
  3. 补上代价。

不要硬套英文博客常见的“抽象观点 -> 举例 -> 升华”套路。

少靠连接词制造逻辑感。
优先让逻辑靠事实顺序和对象关系成立，不靠一串连接词硬拽。

这些词不是不能用，但不要太密：

  1. 因此
  2. 然而
  3. 与此同时
  4. 进一步而言
  5. 换言之
  6. 某种程度上
  7. 从另一个角度看

6. 不要写成站在台上点评系统

少用这些语气词去硬装经验感：

  1. 其实
  2. 当然
  3. 通常
  4. 真正
  5. 这也是为什么
  6. 最值钱的是
  7. 我后来反感的

经验感不是靠这些词堆出来的。
经验感来自具体背景、具体取舍、具体代价。

补充：不要把技术段写成术语瀑布

技术文章不是术语越多越专业。
当技术密度过高时，最常见的问题不是“内容太深”，而是“信息类型混在一起，读者没有抓手”。

重点清理这些情况：

  1. 连续报组件名、机制名、缩写，但对象关系没有立起来。
  2. 一段里同时讲背景、原理、实现、边界、延伸，导致主线被淹没。
  3. 重要判断前堆了太多定义和旁支，读者还没走到结论就已经掉线。
  4. 为了显得专业，把本来一句能讲清的动作写成一串抽象名词。

默认处理原则：

  1. 一段只承载一种主要信息功能：背景、判断、机制、代价，尽量不要全塞进去。
  2. 术语第一次出现时，要么立刻解释它在当前问题里干什么，要么先别写。
  3. 如果技术细节会打断主线，就把它压短，只保留支撑当前判断必需的部分。
  4. 读者需要记住的不是术语数量，而是对象关系和关键动作。

7. 保留人味，不装人味

建议把句子重心放在两类内容上：

  1. 决策句：约束、取舍、动作、结果。
  2. 叙事句：误判、转折、信息不足。

不用硬抠比例，但默认应当以决策句为主，叙事句补足真实过程。

每篇文章尽量至少保留一处真实人类痕迹，例如：

  1. 我原本以为……后来发现……
  2. 当时的错误判断是……修正动作是……
  3. 目前仍不确定的是……

重点是让真实犹豫可见，不是靠口语词和情绪词假装有人味。

8. 接受必要的语言毛边

不要把所有句子都修成同样长度、同样节奏、同样饱满。

默认接受这些情况：

  1. 局部重复。
  2. 轻微停顿。
  3. 转折不那么工整。
  4. 某些句子比较硬、比较笨，但信息量更高。

有信息量的笨句，通常优先于没信息量的漂亮句。
不要为了“更像文章”就把原文里有判断力的硬句子抹平。

去 AI 味不是去掉一切棱角，而是保留真实写作里的不齐整。

9. 可以写感受，但不要写成空情绪

允许出现个人感受，例如：

  1. 我当时不太敢继续这么推。
  2. 这一步开始让我觉得系统在变重。
  3. 我后来对这种做法越来越别扭。

但感受后面最好接具体原因：

  1. 是故障域变大了。
  2. 还是状态散了。
  3. 还是迁移太难了。

感受不是用来制造人味的，要和真实判断、误判、压力、取舍绑在一起。

少写悬空情绪句，例如：

  1. 这让我深受启发
  2. 我也更加坚定了一个想法
  3. 令人遗憾的是

优先保留带原因的主观看法，少写无对象的态度表达。

10. 保留作者偏好

如果作者明显对某种方案有偏好，不要强行改成四平八稳。

只要这种偏好有边界、有理由，就应该保留：

  1. 作者偏向哪种做法。
  2. 为什么偏向。
  3. 代价是什么。
  4. 哪些场景下不成立。

不要为了“客观”把鲜明判断都改成温吞结论。
技术博客允许作者有立场，前提是立场落在具体约束和代价上。

11. 保留作者指纹

保留作者常用的推进方式，不强行换成统一腔调。

默认优先保留这些东西：

  1. 作者反复盯的那几个核心词。
  2. 作者原本的偏好、犹豫和局部执念。
  3. 作者自己的判断路径。
  4. 某些不那么漂亮、但很有辨识度的句子组织方式。

不要把所有句子都改成“更好看”的版本，结果把人写没了。
如果原文有明显的个人判断路径，优先保留路径，不优先追求语言对称。

改写顺序

1. 先把背景立住

起手不要掉进技术名词。
除非文章本来就是组件深挖，否则开头优先回答：业务在做什么、当时卡住什么、为什么这个问题值得写。

开头至少要回答这些问题：

  1. 当时系统处在什么阶段。
  2. 哪些前提已经定了。
  3. 现在在讨论什么。
  4. 真正卡住什么。

如果这是独立文章，开头必须能独立成立。
如果这是承接上文的文章，就直接写清承接关系。

不要把“当时”写成空时间词。
写“当时”时，至少要落到：哪些前提已定、这次只讨论哪一层、现实卡点是什么。

如果文章是在既定约束下做选择，要把约束写出来。
例如：topic 已经按租户拆好、消费侧已经在线上跑着、现在不是重做 topic，而是在讨论后续怎么接。

段落默认优先写成这个顺序：

  1. 当时系统是什么样。
  2. 具体卡在哪里。
  3. 为什么这样改。
  4. 这个改法没解决什么。

2. 再把抽象词压回具体对象

不要把工具写成起点。
正常顺序是：先有系统问题，再有方案设想，最后才到 etcd / zk / Pulsar / Router 这种落地手段。

优先把这些词拆开：

  1. 承接
  2. 分层
  3. 治理
  4. 策略
  5. 复杂度
  6. 能力
  7. 模型

少写这类悬空表达：

  1. 进入消费模型
  2. 承接关系
  3. 形成稳定分层
  4. 职责下放
  5. 复杂度回到主问题

如果必须写，后面必须立刻补一句具体系统状态或动作。

能改成具体动作就改成具体动作。
比如“消费者怎么订阅、资源怎么分、哪个租户出问题怎么处理”，就不要只写“大租户治理”和“资源承接”。

不要制造假洞察。
少写这些句子：

  1. 这背后反映的是……
  2. 更深层的问题在于……
  3. 归根结底，这是一个……问题

如果要上升抽象，必须立刻落回具体对象和动作。
不能只把原文换个更大的词再说一遍。

3. 再把技术细节写实

不要拿空公式和泛估算冒充技术深度。
优先写这些具体东西：

  1. 连接
  2. 接收队列
  3. ack 状态
  4. cursor
  5. metadata
  6. backlog
  7. 实例重启
  8. 恢复一轮要多久

技术真实感不等于术语密度。
不要只写 Pulsar、subscription、router、shard 这些词。

更重要的是写清：

  1. 哪个状态多起来了。
  2. 哪个恢复动作开始变慢。
  3. 哪类故障开始从点状变片状。
  4. 哪种配置开始越来越难改。

技术感来自对象 + 变化 + 代价，不是来自组件名堆叠。
如果一句话只是在报术语，没有说对象关系和代价，优先重写。

如果一段里的技术词很多，但删掉一半不影响核心判断，通常说明技术密度已经过高。
这时优先减的是堆叠和旁支，不是支撑判断所必需的硬内容。

4. 最后才修语气

修语气时只做这些事：

  1. 去掉端着的句型。
  2. 去掉“不是 A，而是 B”这种对抗句。
  3. 去掉对读者说话。
  4. 去掉解释写法的句子。
  5. 去掉“这里把事实说准一点”“回头看我更相信的是”“如果我要判断”这种抬高语气的句子。
  6. 去掉“先把……”“先看……”“真正值得写的是……”这种方法论起手。
  7. 不要为了修齐句子，把原文的硬判断、毛边和偏好一起抹平。

不要写这些东西：

  1. 元说明腔：这篇如果不先……、先把问题问对、不然整篇会写偏
  2. 解释写法的句子：这篇怎么写、这一段为什么先写
  3. 八股结构：为了凑“背景-意义-定义-步骤-总结”而牺牲自然表达

再补一组默认清理项：

  1. 空泛判断：提升效率、最佳实践、值得借鉴、显著优化
  2. 无条件结论：应该、必须、一定，但没写边界
  3. 术语堆叠：连续名词很多，但没有场景约束
  4. 只讲成功：没有代价、失败模式、回滚点
  5. 伪装人味：刻意口语化、刻意错别字、情绪词堆砌

5. 结尾不要硬收束

不是每篇都要写出完整结论、统一答案、方法论升华。

如果问题本来还在演进，更适合写清楚：

  1. 现在先这样做。
  2. 哪些地方以后还可能改。
  3. 哪些判断目前还不稳定。

比起硬写一个 最后我们得出，更重要的是把当前停点写准。

6. 不要把文章补得过度完整

如果原文只是想讲一个局部判断，不要主动补成全景方法论。

默认不要做这些事：

  1. 作者没有定义术语，却自动替作者补一套术语定义。
  2. 原文重点是某个转折，却为了完整性补太多背景知识。
  3. 作者没打算展开的支线，被补成并行章节。
  4. 只想回答一个具体问题的文章，被改成讲完整体系。

允许文章只回答一个具体问题，而不是把整套体系都讲完。

删改边界

  1. 如果一段承载了真实背景、真实犹豫、真实 trade-off、真实建模分叉，优先重写表达，不要先删。
  2. 只有一段既没有新信息，也没有真实判断，只是在重复包装时，才允许直接删。
  3. 不要为了去 AI 味而过度精简。重点不是短，而是有料。
  4. 现实优先。要保留时间、人手、成本、兼容性、历史包袱、团队习惯、风险这些现实限制。
  5. 如果一段读起来像“站在台上点评系统”，而不是“在复盘当时怎么想”，优先压回日常表达。
  6. 如果一段有明显作者偏好、真实毛边、真实情绪，但这些东西都挂在具体原因上，优先保留，不要修成中性正确。
  7. 如果作者原文里有模糊感、试探感、阶段性感，不要替作者做最终拍板。

允许局部不稳

允许写：

  1. 我当时还拿不准。
  2. 这一步先这么做。
  3. 后面再看要不要继续推。

技术文章可以有未决问题，不需要每次都变成结论。
不要把试探性判断改成确定性结论。

默认输出

使用这个 skill 时，默认输出三部分：

  1. 文风问题清单：
    • 哪些地方像模型在总结。
    • 哪些地方背景没立住。
    • 哪些地方说教味或元说明腔太重。
    • 哪些地方技术细节太空。
  2. AI 味来源诊断：
    • 这段像 AI 的主要原因是什么。
    • 是抽象过高、推进太齐、假洞察，还是语气太端。
    • 改写时保留了什么：背景、犹豫、硬判断，还是作者偏好。
  3. 改写后的版本：
    • 保留作者原本判断。
    • 压低抽象度。
    • 压平语气。
    • 不擅自补作者没说过的经历。

持续记录那些真正影响结果的工程取舍。
