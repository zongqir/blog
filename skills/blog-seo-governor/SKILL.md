# 博客 SEO 治理

## 核心职责

这个 skill 统一处理文章发现能力，不只看单篇 `frontmatter`。

它主要处理五类问题：

1. 文章级 SEO 元数据是否完整、可读、可被检索。
2. 标签体系是否稳定，是否能支撑归档、聚合和相关文章推荐。
3. `series`、`featured`、`hot` 这些运营字段是否自洽，是否能驱动首页模块。
4. 内部链接、标签页、归档页、相关文章链路是否形成可发现闭环。
5. 发布前 SEO 门禁是否过线。
6. SEO 相关分发入口是否违反站点视觉系统约束。

它默认不处理这些事：

1. 大纲是否顺。
2. 文风是否自然。
3. 正文事实是否成立。
4. 整站视觉改版或纯审美层面的 UI 设计。

补充边界：

1. 这个 skill 不负责重新做视觉方案。
2. 但它要检查 SEO 与分发入口是否因为运营诉求破坏视觉系统约束，例如：首页卡片层级失衡、标签页信息组织失真、归档或相关文章区域出现硬编码视觉特例。

如果问题主要是结构和标题推进，优先用 `$blog-outline-doctor`。  
如果问题主要是正文表达和去 AI 味，优先用 `$blog-writing-style`。  
如果问题主要是文章整体是否可发布，可以配合 `$blog-quality-gate` 一起用。

## 处理范围

### 1. 文章级

默认覆盖这些字段和对象：

1. `title`
2. `description`
3. `date`
4. `updated`
5. `tags`
6. `series`
7. `series_part`
8. `series_intent`
9. `series_buildable`
10. `featured`
11. `featured_rank`
12. `hot`
13. `hot_rank`
14. `draft`
15. `canonical_url`
16. permalink / slug
17. 正文内部链接

### 2. 站点级

默认覆盖这些聚合与分发点：

1. 首页精选
2. 首页热门
3. 专题连载
4. 标签页
5. 归档页
6. 相关文章推荐
7. 上述入口的信息层级与视觉一致性

### 3. 发布前门禁

默认覆盖这些检查项：

1. 元数据完整性
2. 标签与专题一致性
3. 链接与锚点可达性
4. 首页卡片字段完整性
5. 相关文章最小数量要求
6. 构建与关键页面检查

## 默认工作顺序

处理 SEO 时，按这个顺序走，不要跳步：

1. 先看文章级元数据。
2. 再看标签和专题归类。
3. 再看 slug、内部链接、相关文章。
4. 再看首页聚合与运营位。
5. 最后给发布门禁结论。

原因很简单：  
如果最基础的标题、摘要、标签都不稳，后面的聚合与推荐判断也会一起发飘。

## 文章级 SEO 规则

### 1. `title`

默认要求：

1. 能表达具体问题，而不是空泛口号。
2. 不要只堆术语。
3. 和正文真实内容一致。
4. 不要为了 SEO 硬塞关键词，牺牲可读性。

### 2. `description`

默认要求：

1. 建议 50 到 120 字。
2. 要能独立解释“这篇在讲什么、为什么值得看”。
3. 不要只重复标题。
4. 不要写成营销话术。

### 3. slug / permalink

默认要求：

1. 语义稳定。
2. 和标题主题一致。
3. 不要频繁改历史文章 slug。
4. 需要变更时，先确认旧链接影响。

### 4. `date` 与 `updated`

默认要求：

1. `date` 只表示首次发布时间。
2. `updated` 只在文章有真实修改时更新。
3. 不要拿 `updated` 制造“看起来常更新”的假象。

### 5. `canonical_url`

默认要求：

1. 只在转载、多站同步、需要规范化时填写。
2. 没有明确规范化需求时，不要硬加。

## 标签体系规则

### 1. 标签的职责

标签不是关键词垃圾桶。  
它至少要承担三件事：

1. 表达主题。
2. 支撑标签页聚合。
3. 为相关文章推荐提供回退依据。

### 2. 标签数量

默认 `2` 到 `6` 个。  
少于 `2` 个，主题表达容易失真。  
多于 `6` 个，通常说明把相关词都当成标签塞进去了。

### 3. 自动标签映射

标签按标题、摘要、正文关键词综合判断，不只看标题。

默认映射：

1. `国际化`：国际化、i18n、locale、多语言、时区、UTC。
2. `架构设计`：架构、DDD、领域、状态机、适配器、边界。
3. `工程治理`：治理、规范、门禁、checklist、流程、标准化、自动化。
4. `SRE`：故障、告警、监控、可用性、稳定性、恢复、容量。
5. `性能优化`：性能、延迟、吞吐、CPU、内存、IO、GC、压测。
6. `Linux`：Linux、top、vmstat、pidstat、iostat、内核、系统调用。
7. `ToB`：ToB、B端、企业客户、SaaS。

### 4. 打标签原则

1. 至少保留一个主主题标签。
2. 正文和标题信号冲突时，优先以正文实际内容为准。
3. 不要为了覆盖搜索词，把所有相关词都变成标签。
4. 没有足够证据时，直接标为待确认，不要瞎补。

## 专题与运营字段

### 1. `series`

默认映射：

1. 命中 `国际化` => `series: "全球化工程治理"`。
2. 命中 `SRE` 或 `Linux` => `series: "SRE 故障实战"`。
3. 命中 `架构设计` 或 `工程治理` => `series: "工程治理体系"`。

### 2. `series_buildable`

默认规则：

1. 命中系列映射时，默认 `true`。
2. 未命中时，默认 `false`。
3. 如果作者明确说不是系列文，可以手动改成 `false`。

### 3. `series_part`

只有在 `series_buildable=true` 时才生成。

生成方式：

1. 扫描 `_posts/` 中同系列历史文章。
2. 取现有最大 `series_part + 1`。
3. 若没有历史文章，从 `1` 开始。

### 4. `series_intent`

默认模板：

1. `全球化工程治理` => `围绕国际化、多语言、时区与跨区域交付中的工程约束，整理一套可落地的治理方法。`
2. `SRE 故障实战` => `围绕故障定位、容量判断、恢复流程与稳定性建设，沉淀可复用的 SRE 实战经验。`
3. `工程治理体系` => `围绕架构演进、工程规范、流程门禁与长期维护成本，整理一套可持续的工程治理方法。`

### 5. `hot`

默认规则：

1. 命中 `架构设计`、`工程治理`、`SRE`、`性能优化` 任一主题信号时，默认 `hot: true`。
2. 若只命中兜底标签，默认 `hot: false`。

### 6. `hot_rank`

默认粗排：

1. `架构设计` => `10`
2. `工程治理` => `20`
3. `SRE` => `30`
4. `性能优化` => `40`

如果命中多个主题，取更小的那个值。  
后续允许手动微调。

### 7. `featured`

`featured` 不是自动推断字段。  
默认只在作者明确要求首页精选，或者已有明确运营判断时才填写。

## 首页与聚合规则

首页模块必须由 `frontmatter` 驱动，不允许硬编码文章 URL。

默认规则：

1. 精选文章：`featured=true`，按 `featured_rank` 升序，再按 `updated` 降序。
2. 热门文章：`hot=true`，按 `hot_rank` 升序，再按 `updated` 降序。
3. 专题连载：按 `series` 聚合；缺失时可以回退到约定标签映射。
4. 状态条只展示真实可核查指标，例如总文章数、最近更新时间、专题数量。

默认原则：

1. 首页不展示虚构访问量。
2. 热度感优先通过 `热门`、`精选`、`连载` 这些标签体系表达。
3. `draft=true` 的文章不得进入首页运营位。
4. 首页、标签页、归档页、相关文章这些 SEO 入口，不得为了关键词覆盖或点击刺激破坏视觉 token、阅读节奏和既有信息层级。

## 视觉系统约束

这个 skill 需要同时检查 SEO 分发入口是否遵守视觉系统约束。  
原因不是“视觉归 SEO 管”，而是首页、标签页、归档页、相关文章这些入口本来就是搜索与分发链路的一部分。

### 1. 允许优先调整

这些调整默认允许做：

1. 颜色 token：`--text`、`--muted`、`--accent`、`--line`
2. 节奏 token：字号层级、行高、间距、卡片内边距
3. 交互强度：hover、阴影、边框对比、动效时长

### 2. 谨慎调整

这些调整需要明确理由，不能为了 SEO 或点击率随手改：

1. 布局结构：模块顺序、信息架构、导航结构
2. 文章页阅读路径：封面、目录、正文宽度核心逻辑
3. 标签页与归档页的数据组织方式

### 3. 禁止事项

这些情况直接判为违规：

1. 为追求“炫酷”引入影响阅读的重动效与噪点背景
2. 无依据地切换整体视觉风格，导致站点品牌漂移
3. 在未定义 token 的情况下散落硬编码色值与尺寸
4. 为了关键词覆盖或卡片点击率，在首页、标签页、归档页、相关文章区域临时塞入一次性样式和视觉特例

### 4. 判断原则

1. 不以“视觉好看”为理由破坏内容真实性。
2. 不以“SEO 更强”为理由破坏阅读路径和信息层级。
3. 不以“运营曝光”为理由制造硬编码视觉例外。

## 内部链接与相关文章

### 1. 内部链接

检查这些事：

1. 新文是否有合理的内部链接入口或出口。
2. 标签锚点、目录锚点、文章 permalink 是否可跳转。
3. 不要给出失效链接。

### 2. 相关文章推荐

默认生成顺序：

1. 先按 `series` 推荐。
2. 同系列不足时，回退按标签重叠度补足。
3. 默认展示 `3` 篇。

质量规则：

1. 排除当前文章。
2. 排除 `draft=true` 文章。
3. 发布前至少满足“可推荐数量 >= 3”。
4. 推荐结果尽量覆盖“至少 1 篇基础向 + 1 篇进阶向”。

## 发布门禁

### 1. 内容与渲染门禁

1. `bundle exec jekyll build` 构建通过。
2. 关键页面渲染正常：`/`、`/archive/`、`/tags/`、`/about/`。
3. 新文 permalink、标签锚点、目录锚点可跳转。
4. 首页精选、热门、专题卡片无空字段、无死链接。

### 2. SEO 与运营门禁

1. `frontmatter` 字段完整且格式合法。
2. `title`、`description`、slug、tag、内部链接一致。
3. 无明显硬编码运营数据，例如虚构访问量、伪造统计。
4. 专栏评估字段合法：`series_buildable`、`series_part`、`series_intent` 一致。
5. 相关文章推荐满足最小数量与质量规则。
6. 首页、标签页、归档页、相关文章等 SEO 分发入口符合视觉系统约束，不出现破坏阅读路径的硬编码样式和信息层级异常。
7. 推送后 GitHub Actions 的 Pages 工作流成功。

## 默认输出

使用这个 skill 时，默认输出四部分：

1. SEO 问题清单：
   - 缺了什么字段。
   - 哪些标签或运营字段冲突。
   - 哪些链接与聚合链路有问题。
2. 推荐的元数据：
   - 给出可直接落回文章文件的 YAML。
3. 站点分发建议：
   - 标签、专题、相关文章、首页曝光位应该怎么挂。
4. 门禁结论：
   - 可发布 / 接近可发布 / 明显不过线。
5. 视觉约束结论：
   - SEO 分发入口是否符合视觉系统约束，若不符合，指出具体入口和破坏点。

## 红线

1. 不要伪造描述、热度、访问量、统计或外链来源。
2. 不要把不稳定推断写成确定事实。
3. 不要为了 SEO 堆砌标题、摘要和标签。
4. 不要让 `draft`、`featured`、`hot`、`series_buildable` 这些状态彼此打架。
5. 不要把这个 skill 用成正文润色器，它负责的是发现、聚合和发布治理。
6. 不要为了 SEO 或运营曝光，给首页、标签页、归档页、相关文章区域塞破坏视觉系统的一次性样式和信息堆叠。
