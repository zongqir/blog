博客写透度审查

核心定位

这个 skill 不负责把文章写短、写顺、写漂亮。
它负责判断：哪里没写透，以及哪里应该直接补写。

它主要做四件事：

  1. 找出文章里最难懂、最薄、最容易让人跟丢的地方。
  2. 判断这些地方是“该展开”还是“只是啰嗦”。
  3. 直接把关键段落写透，而不是只给抽象建议。
  4. 保留作者原本判断，不把补写变成另一篇文章。

它不是总检，不是文风统一器，也不是大纲重排器。

这个 skill 解决什么问题

适合处理这些情况：

  1. 文章不算错，但读者看不太懂作者到底在说什么。
  2. 结论有了，但依据太薄。
  3. 系统状态不可见，只剩抽象判断。
  4. 代价、边界、故障面被点到为止，没有展开。
  5. trade-off 写了名字，但没有落到现实动作和代价。
  6. AI 在改稿时把人的真实表达压短了、抹平了、整理过头了。
  7. 技术密度明显不足，读者只能看到术语或结论，看不到对象、状态、动作和代价。
  8. 技术密度分配错位，简单部分写得很满，真正复杂和值钱的技术判断反而写得很轻。

默认边界

这个 skill 默认不处理这些问题：

  1. 二级三级标题怎么重排。
  2. 文风怎么更自然、更有人味。
  3. 是否存在 AI 味或翻译腔。
  4. 是否可发布。
  5. frontmatter、SEO、标签、专题和首页聚合。

如果问题主要是结构，优先用 $blog-outline-doctor。
如果问题主要是文风，优先用 $blog-writing-style。
如果问题主要是总检分诊，优先用 $blog-quality-gate。

核心判断

1. 先区分“没写透”和“只是长”

不要默认“多几句就是啰嗦”。
先判断这些句子是不是在补关键内容：

  1. 判断依据。
  2. 系统状态。
  3. 触发条件。
  4. 代价出现的方式。
  5. 没解决什么。

如果一段在补这些东西，它就不是废话。
如果只是换一种说法重复原结论，才算冗余。

2. 找“该展开的地方”

最常见的没写透位置有这些：

  1. 只给结论，不给为什么这样判断。
  2. 只写“复杂度变高了”，不写哪个状态开始多、哪个动作开始慢。
  3. 只写“故障域变大了”，不写到底从一个点怎么扩成一片。
  4. 只写“后面会很难维护”，不写难在配置、迁移、接管、排查还是恢复。
  5. 只写“trade-off”，不写两边分别牺牲了什么、保住了什么。
  6. 只写“这里我开始别扭了”，不写别扭感是被什么现实触发的。

3. 写透不等于写长

写透的目标不是把所有东西都铺开。
目标是让读者跟得上作者的判断路径。

优先补这些内容：

  1. 现实触发点。
  2. 对象关系。
  3. 状态变化。
  4. 代价出现方式。
  5. 当前改法没解决什么。

不要为了“完整”自动补这些东西：

  1. 与当前问题无关的百科背景。
  2. 作者没打算展开的支线。
  3. 一整套方法论包装。
  4. 并不服务当前判断的大段定义。

4. 笔墨要向最值钱的地方倾斜

不要把篇幅平均分给每个部分。
技术文章不是信息越多越显得有水平，而是最关键的判断有没有被写透。

重点看这些失衡信号：

  1. 简单背景和常识性铺垫写了很多，真正复杂的判断只带过一句。
  2. 知识点、概念点很多，但没有一个被真正展开成强洞察。
  3. 容易写的内容写得很满，真正有代价、有取舍的地方写得很轻。
  4. 段落看起来都不短，但篇幅没有明显向高价值判断倾斜。

如果读者读完只觉得“信息很多”，却说不出文章最强的一个判断，通常不是内容不够，而是笔墨花错了地方。

默认处理原则：

  1. 简单内容写到够支撑后文判断为止，不要反复铺陈。
  2. 把篇幅优先留给最关键的判断依据、系统状态变化、代价落点和边界。
  3. 遇到“简单内容写很多，复杂内容写不够”，优先判断为写透度问题，不要先当作文风问题处理。
  4. 不要为了段落均匀、结构工整或信息看起来很多，就把真正值钱的部分压短。

5. 技术密度不够，通常不是“少写点术语”能解决

技术密度不足，不是指文章里技术名词少。
更常见的情况是：名词有了，但技术判断还没有落到具体对象和变化上。

重点看这些信号：

  1. 写了组件名、机制名、模式名，但没写它在当前场景里到底解决什么问题。
  2. 写了“复杂”“隔离”“治理”“稳定性”这类词，但没写哪个状态变了、哪个动作变慢了、哪个对象被影响了。
  3. 一段里有结论，但没有最基本的判断依据，读者只能被迫相信作者。
  4. 技术段落读起来像摘要，信息类型只有名词，没有关系、动作和代价。

默认补写方向：

  1. 把抽象词压回对象。
  2. 把对象补成关系。
  3. 把关系补成动作。
  4. 把动作补上代价和边界。

写透检查维度

1. 判断路径有没有写出来

重点看：

  1. 作者当时为什么会那样想。
  2. 哪一步开始改变了原来的判断。
  3. 后来为什么不愿意继续往那条路推。

如果只剩结论，没有判断路径，通常就是没写透。

2. 系统状态有没有可见

重点看：

  1. 当时系统是什么样。
  2. 哪个状态开始变多。
  3. 哪个动作开始变慢。
  4. 哪个环节开始变难改、难迁、难接管、难排查。

只写抽象名词，不写系统状态，通常会让技术文章发空。

3. 触发点有没有出现

重点看：

  1. 作者为什么会在那个时间点开始重新看这个问题。
  2. 是租户数起来了，还是流量分布变了，还是故障模式变了。
  3. 是不是有一个现实压力把问题从“能跑”逼成“必须回答”。

没有触发点，很多判断都会像凭空冒出来。

4. 代价有没有展开

重点看：

  1. 代价到底落在什么对象上。
  2. 是资源利用率差、实例数膨胀、迁移难、恢复慢，还是值班复杂度上升。
  3. 是短期代价，还是长期维护代价。

只写“有代价”“会更复杂”，不算写透。

5. 没解决什么有没有写

重点看：

  1. 这个改法解决了什么。
  2. 它没解决什么。
  3. 它只是把问题挪到了哪。

如果只写优点，不写没解决什么，文章通常会显得薄。

6. 技术密度是不是落在了该落的地方

重点看：

  1. 技术细节是不是都堆在容易写的背景段，而关键判断本身仍然是薄的。
  2. 真正决定结论成立的那几个机制、状态变化、代价落点，是否得到了足够篇幅。
  3. 是不是讲了很多周边知识，但对当前文章的主判断帮助很小。
  4. 读者读完后，能不能说清文章里最关键的技术判断凭什么成立。

如果技术信息很多，但关键判断仍然站不住，通常不是“信息量不足”，而是技术密度落错了地方。

直接补写的规则

1. 只补关键段

默认补 1 到 5 处最关键的位置，不整篇泛扩。

优先补：

  1. 开头没立住的关键背景。
  2. 中间最薄的判断依据。
  3. 最需要展开的系统状态和代价。
  4. 结尾本该补上的当前边界或未解决问题。

2. 补写时保留作者原意

补写不是重新发明观点。
只能做这些事：

  1. 把原本已经存在的判断说透。
  2. 把被压缩掉的因果链补出来。
  3. 把抽象词压回对象、状态、动作、代价。

不要做这些事：

  1. 擅自改变文章主类型。
  2. 擅自追加作者没表达过的新立场。
  3. 把局部判断补成一整套体系。
  4. 替作者做他原文里没有做出的最终拍板。

3. 优先补“为什么”和“怎么变重”

最优先补的通常不是术语定义，而是：

  1. 为什么作者开始这样判断。
  2. 系统是怎么一点点变重的。
  3. 代价是怎么落到现实链路上的。
  4. 这个改法为什么仍然不够。

默认输出

使用这个 skill 时，默认输出三部分：

  1. 没写透的位置：
    • 指出 1 到 5 个最关键的薄点。
    • 说明这里薄在判断依据、系统状态、触发点、代价，还是边界。
  2. 为什么这里不能再压：
    • 说明这不是废话，而是读者理解判断所必需的信息。
  3. 直接补写后的版本：
    • 直接把关键段写透。
    • 保留作者原意。
    • 不擅自改文章类型和立场。

红线

  1. 不要把“写透”误做成“写长”。
  2. 不要把局部判断强行补成全景方法论。
  3. 不要为了自洽补作者没打算展开的支线。
  4. 不要把作者试探性的判断改成确定结论。
  5. 不要把这个 skill 用成文风美化器，它负责的是补内容厚度。
