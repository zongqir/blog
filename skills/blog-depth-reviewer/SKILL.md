# 博客写透度审查

## 核心定位

这个 skill 不负责把文章写短、写顺、写漂亮。  
它负责判断：哪里没写透，以及哪里应该直接补写。

它主要做五件事：

1. 找出文章里最难懂、最薄、最容易让人跟丢的地方。
2. 判断这些地方是“该展开”还是“只是啰嗦”。
3. 直接把关键段落写透，而不是只给抽象建议。
4. 保留作者原本判断，不把补写变成另一篇文章。
5. 看到明显未完成标记时，优先把它补成正文，而不是直接清掉标记了事。
6. 如果用户已经点名“把 A 文里的某些抓手塞进 B 文”，优先把用户点名的内容显性写进 B 文，而不是先迁移模型自己挑的抽象层。
7. 如果文章在讲治理、可观测性、机制建设、平台能力，默认检查它是不是只写了“平台怎么建”，却没写“业务怎么用、为什么现场还是用不上”。

它不是总检，不是文风统一器，也不是大纲重排器。

## 这个 skill 解决什么问题

适合处理这些情况：

1. 文章不算错，但读者看不太懂作者到底在说什么。
2. 结论有了，但依据太薄。
3. 系统状态不可见，只剩抽象判断。
4. 代价、边界、故障面被点到为止，没有展开。
5. trade-off 写了名字，但没有落到现实动作和代价。
6. AI 在改稿时把人的真实表达压短了、抹平了、整理过头了。
7. 技术密度明显不足，读者只能看到术语或结论，看不到对象、状态、动作和代价。
8. 技术密度分配错位，简单部分写得很满，真正复杂和值钱的技术判断反而写得很轻。
9. 草稿里留着 `TODO`、作者口语批注、转写碎句或“这里没写完/需要优化”之类标记，说明那一段其实还没完成。
10. 作者把内容压得太凝练，结论虽然都对，但读者跟不上中间判断台阶。

## 默认边界

这个 skill 默认不处理这些问题：

1. 二级三级标题怎么重排。
2. 文风怎么更自然、更有人味。
3. 是否存在 AI 味或翻译腔。
4. 是否可发布。
5. frontmatter、SEO、标签、专题和首页聚合。

如果问题主要是结构，优先用 `$blog-outline-doctor`。  
如果问题主要是文风，优先用 `$blog-writing-style`。  
如果问题主要是总检分诊，优先用 `$blog-quality-gate`。

再补一个强触发：

1. 如果当前文章里已经出现 `TODO`、`待补`、`需要优化`、作者口语批注或显式“这里没说清楚/还没写完”，即使用户这轮没有明确说“写深一点”，也默认应当触发本 skill。
2. 如果用户说“按我的评论改”“按批注改”“按我留的那些评论优化”，默认先把评论逐条翻译成补写任务，再进入正文修改；不能跳过评论直接自由补写。
3. 如果正文里已经有 `事实性错误`、`AI痕迹重`、`很怪`、`不像人写` 等作者强批注，默认先修这些位置，再扩写其他部分。

## 核心判断

### 1. 先区分“没写透”和“只是长”

不要默认“多几句就是啰嗦”。  
先判断这些句子是不是在补关键内容：

1. 判断依据。
2. 系统状态。
3. 触发条件。
4. 代价出现的方式。
5. 没解决什么。
6. 读者理解这个判断所必需的中间台阶。

如果一段在补这些东西，它就不是废话。  
如果只是换一种说法重复原结论，才算冗余。

再补一个常见误判：

1. 不要把“语言很凝练”自动当成“表达很好”。
2. 如果读者需要自己补前提、补对象、补因果，这通常不是高级，而是没写透。

### 2. 找“该展开的地方”

最常见的没写透位置有这些：

1. 只给结论，不给为什么这样判断。
2. 只写“复杂度变高了”，不写哪个状态开始多、哪个动作开始慢。
3. 只写“故障域变大了”，不写到底从一个点怎么扩成一片。
4. 只写“后面会很难维护”，不写难在配置、迁移、接管、排查还是恢复。
5. 只写“trade-off”，不写两边分别牺牲了什么、保住了什么。
6. 只写“这里我开始别扭了”，不写别扭感是被什么现实触发的。
7. 结尾开始谈“怎么做”了，但只停在抽象原则，没有把参与对象、责任拆分、验证轮次、灰度或回滚这些现实动作落下来。
8. 方案看起来合理，但默认依赖理想前提，例如默认总能找到懂历史逻辑的 owner、默认灰度在所有交付形态都成立、默认单一维度分析就够了。
9. 方案表面上在降风险，实际上把技术问题翻译成更多流程动作，例如填表、认领 owner、加会议确认，但没有增加自动化回归、环境能力或可执行保护。
10. 一线眼前的问题明明是“不知道漏了哪些场景”，方案却一上来就给防腐层、插件化、配置即代码这类高门槛治理动作，没有先回答怎么低成本识别风险、收小范围和建立最小验证集。
11. 文章已经开始讲“怎么治理”，但没有先定义核心业务对象、业务边界和业务承诺，导致后面的结果指标、流程动作和门禁都飘在半空。
12. 文章要求业务 owner 参与，却没有写清合理分工边界，容易被读成“把平台复杂度甩给业务”。
13. 文章涉及本地化交付转 SaaS、项目制转产品化、私有化转共享化等阶段变化，但没写清旧形态、变更目标和当时最优先验证的商业或交付目标。
14. 提到一个待设计系统或机制，例如调度器、Router、控制面、分桶层，却没有交代它最小怎么工作，导致读者知道“作者想做它”，但不知道“为什么会长出这个想法”。
15. 已经提到方案评审、争论、试运行、值班或迁移现场，但没有把这些场景写开，结果整段只剩事后回顾式抽象判断。
16. 已经给出一套方案轮廓，但没有把评审时真正被追问的关键问题写出来，导致方案像漂在半空，读者没法“打靶”。
17. 文章说“后来觉得这像另一套产品”“主线被带偏了”，却没有写清这种感觉是在什么问题被一层层追出来的。
18. 标题或正文已经承诺“差点做了一套 X 系统 / 原本准备这么做”，但正文没有交代这套东西最小要由哪些组件组成、核心流程怎么走，读者只能看到态度，看不到方案本体。
19. 用户已经明确指出“这段很虚”“没有技术感”“你没说具体怎么做”，但正文还停在组件名和抽象流程，没有写清状态是怎么被感知、结果是怎么传播、执行器怎么动作。

### 3. 写透不等于写长

写透的目标不是把所有东西都铺开。  
目标是让读者跟得上作者的判断路径。

优先补这些内容：

1. 现实触发点。
2. 对象关系。
3. 状态变化。
4. 代价出现方式。
5. 当前改法没解决什么。
6. 这个改法成立所依赖的前提，在现实里是否真的存在。
7. 如果主题是指标、监控、治理、平台能力，补“业务 owner 为什么会用/不会用”“真正的低摩擦路径是什么”。
8. 如果原文已经把判断压得太短，优先补出中间因果，不要只做同义改写。
9. 如果文章讨论的是阶段切换，补出旧形态、当前变更目标和为什么那时先追求验证速度或商业价值。
10. 如果文章里冒出了一个新系统，补出它最小工作流：谁看状态、谁做决策、谁执行变更。
11. 如果用户明确说“这里没有做这件事情的感觉”，优先补方案评审、争论、试运行、迁移决策这些现场，而不是继续补更大的抽象总结。
12. 如果作者说“你这段不写清楚，别人怎么打靶”，优先补方案在评审里被挑战的点：怎么算、怎么接管、怎么迁移、哪里可能打架，而不是只补更大的意义。
13. 如果文章真正停在“要不要做这套东西”，优先补决策分叉和挑战问题，不要自动往下脑补 `etcd/zk` 这类实现选型。
14. 如果用户明确要求“多一点现场感”“谁提了不同意见”，默认至少补出两类角色的关注点，以及哪一个追问真正把讨论从“方向对”扳到“主线是否值得做”。拿不准具体人名时，用角色，不要硬编姓名和精确引语。
15. 如果用户指出“你到现在还没说具体怎么做”，默认至少补出最小可运行方案：有哪些角色或组件、接入怎么发生、迁移怎么发生、故障时谁接管。先把系统工作流讲明白，再谈高层判断。
16. 如果用户进一步指出“还是很虚 / 没有技术感”，补写时要继续从组件层往运行机制下压：状态存在哪、靠什么感知变化、谁 watch、谁更新版本、谁执行 drain / handoff / 接管。不要只把抽象流程换个说法再写一遍。

不要为了“完整”自动补这些东西：

1. 与当前问题无关的百科背景。
2. 作者没打算展开的支线。
3. 一整套方法论包装。
4. 并不服务当前判断的大段定义。

### 4. 笔墨要向最值钱的地方倾斜

不要把篇幅平均分给每个部分。  
技术文章不是信息越多越显得有水平，而是最关键的判断有没有被写透。

重点看这些失衡信号：

1. 简单背景和常识性铺垫写了很多，真正复杂的判断只带过一句。
2. 知识点、概念点很多，但没有一个被真正展开成强洞察。
3. 容易写的内容写得很满，真正有代价、有取舍的地方写得很轻。
4. 段落看起来都不短，但篇幅没有明显向高价值判断倾斜。

如果读者读完只觉得“信息很多”，却说不出文章最强的一个判断，通常不是内容不够，而是笔墨花错了地方。

默认处理原则：

1. 简单内容写到够支撑后文判断为止，不要反复铺陈。
2. 把篇幅优先留给最关键的判断依据、系统状态变化、代价落点和边界。
3. 遇到“简单内容写很多，复杂内容写不够”，优先判断为写透度问题，不要先当作文风问题处理。
4. 不要为了段落均匀、结构工整或信息看起来很多，就把真正值钱的部分压短。
5. 如果文章提出的是降风险方案，要主动检查它有没有漏掉组合交互、自动化回归抓手、部署形态差异这些现实约束。
6. 如果方案增加了沟通动作，要追问这些动作到底是临时过渡，还是在替代本该自动化的技术手段。
7. 如果文章越写越像流程手册，优先补回业务触发点、使用摩擦和责任边界，而不是继续加方法论分层。

### 5. 技术密度不够，通常不是“少写点术语”能解决

技术密度不足，不是指文章里技术名词少。  
更常见的情况是：名词有了，但技术判断还没有落到具体对象和变化上。

重点看这些信号：

1. 写了组件名、机制名、模式名，但没写它在当前场景里到底解决什么问题。
2. 写了“复杂”“隔离”“治理”“稳定性”这类词，但没写哪个状态变了、哪个动作变慢了、哪个对象被影响了。
3. 一段里有结论，但没有最基本的判断依据，读者只能被迫相信作者。
4. 技术段落读起来像摘要，信息类型只有名词，没有关系、动作和代价。

默认补写方向：

1. 把抽象词压回对象。
2. 把对象补成关系。
3. 把关系补成动作。
4. 把动作补上代价和边界。

## 写透检查维度

### 1. 判断路径有没有写出来

重点看：

1. 作者当时为什么会那样想。
2. 哪一步开始改变了原来的判断。
3. 后来为什么不愿意继续往那条路推。
4. 如果中间有方案评审、评估或试运行，是否把这些决策时刻写出来了。
5. 如果当时已经有人在追问方案细节，是否把这些追问本身写出来了。

如果只剩结论，没有判断路径，通常就是没写透。

### 2. 系统状态有没有可见

重点看：

1. 当时系统是什么样。
2. 哪个状态开始变多。
3. 哪个动作开始变慢。
4. 哪个环节开始变难改、难迁、难接管、难排查。

只写抽象名词，不写系统状态，通常会让技术文章发空。

### 3. 触发点有没有出现

重点看：

1. 作者为什么会在那个时间点开始重新看这个问题。
2. 是租户数起来了，还是流量分布变了，还是故障模式变了。
3. 是不是有一个现实压力把问题从“能跑”逼成“必须回答”。

没有触发点，很多判断都会像凭空冒出来。

### 4. 代价有没有展开

重点看：

1. 代价到底落在什么对象上。
2. 是资源利用率差、实例数膨胀、迁移难、恢复慢，还是值班复杂度上升。
3. 是短期代价，还是长期维护代价。

只写“有代价”“会更复杂”，不算写透。

### 5. 没解决什么有没有写

重点看：

1. 这个改法解决了什么。
2. 它没解决什么。
3. 它只是把问题挪到了哪。

如果只写优点，不写没解决什么，文章通常会显得薄。

### 6. 技术密度是不是落在了该落的地方

重点看：

1. 技术细节是不是都堆在容易写的背景段，而关键判断本身仍然是薄的。
2. 真正决定结论成立的那几个机制、状态变化、代价落点，是否得到了足够篇幅。
3. 是不是讲了很多周边知识，但对当前文章的主判断帮助很小。
4. 读者读完后，能不能说清文章里最关键的技术判断凭什么成立。

如果技术信息很多，但关键判断仍然站不住，通常不是“信息量不足”，而是技术密度落错了地方。

## 直接补写的规则

### 1. 只补关键段

默认补 1 到 5 处最关键的位置，不整篇泛扩。

优先补：

1. 开头没立住的关键背景。
2. 中间最薄的判断依据。
3. 最需要展开的系统状态和代价。
4. 结尾本该补上的当前边界或未解决问题。
5. 如果文章后半段承诺回答“这类需求该怎么接、怎么完成”，优先补具体接法和收口动作，不要只追加一层更大的原则总结。
6. 如果用户明确点名要把另一篇文章里的具体段落、抓手或判断迁移进来，优先补这些被点名的内容；只有它们显性出现后，才允许补其他延伸判断。
7. 如果用户点名“这里缺背景”“这里没有做事感”“这里没讲清系统怎么工作”，默认按高优先级补写，不要把它们降级成普通措辞优化。
8. 如果用户点名“这里只有论述，没有做事感”，默认至少补一段评审或讨论现场。拿不准具体人名时，也要写清角色和 challenge，不要继续停在抽象判断层。
9. 如果用户点名要“谁支持、谁反对、怎么被挑战”，优先写出角色分歧和桌上问题如何升级，不要只在段尾补一句“于是我开始犹豫”。

### 2. 补写时保留作者原意

补写不是重新发明观点。  
只能做这些事：

1. 把原本已经存在的判断说透。
2. 把被压缩掉的因果链补出来。
3. 把抽象词压回对象、状态、动作、代价。

不要做这些事：

1. 擅自改变文章主类型。
2. 擅自追加作者没表达过的新立场。
3. 把局部判断补成一整套体系。
4. 替作者做他原文里没有做出的最终拍板。
5. 把作者留作“待补内容”的标记当成普通脏文本删除，却没有把那段真正补完。

### 3. 优先补“为什么”和“怎么变重”

最优先补的通常不是术语定义，而是：

1. 为什么作者开始这样判断。
2. 系统是怎么一点点变重的。
3. 代价是怎么落到现实链路上的。
4. 这个改法为什么仍然不够。
5. 它依赖的组织条件或交付条件，为什么在现实里可能不存在。
6. 如果文章在回答“现在先怎么办”，第一步是不是贴近一线成本结构，还是直接跳到了中长期重构。

## 默认输出

使用这个 skill 时，默认输出三部分：

1. 没写透的位置：
   - 指出 1 到 5 个最关键的薄点。
   - 说明这里薄在判断依据、系统状态、触发点、代价，还是边界。
2. 为什么这里不能再压：
   - 说明这不是废话，而是读者理解判断所必需的信息。
3. 直接补写后的版本：
   - 直接把关键段写透。
   - 保留作者原意。
   - 不擅自改文章类型和立场。
4. 若当前任务是“按评论改”或“按批注改”：
   - 先列出评论清单。
   - 标明每条评论将如何处理。
   - 结束前逐条确认是否已经落到正文。

## 红线

1. 不要把“写透”误做成“写长”。
2. 不要把局部判断强行补成全景方法论。
3. 不要为了自洽补作者没打算展开的支线。
4. 不要把作者试探性的判断改成确定结论。
5. 不要把这个 skill 用成文风美化器，它负责的是补内容厚度。
