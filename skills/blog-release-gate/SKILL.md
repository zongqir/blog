# 博客发布流程与门禁

## 核心定位

这个 skill 不负责润色正文。  
它负责把“这篇文章怎么从草稿走到可发布”这条链路走完整。

它主要回答四件事：

1. 现在处在写作流程的哪个阶段。
2. 下一步该做什么，而不是同时做一堆事。
3. 发布前哪些门禁必须过。
4. 页面、链接、构建、聚合位有没有明显问题。

补充参考：

1. `../blog-article-fit-checker/references/write-principles.md` 归档了原始 `write.md`。
2. 其中和发布门禁最相关的是：通用三问、类型化评分、低于及格线时如何降级标注。

## 默认工作顺序

按这个顺序走，不要跳步：

1. 先判断文章当前阶段。
2. 再补齐元数据和站点分发信息。
3. 再做质量总检。
4. 再做 SEO 与聚合检查。
5. 最后做构建、页面核验和发布确认。

不要一上来就问“能不能发”。  
如果前面的阶段还没走完，发布结论没有意义。

再补一条执行约束：

1. 如果用户已经明确说过不要在本机跑 `bundle` / `jekyll build`，或者本地没有 Ruby / Bundler / Jekyll 环境，不要再重复尝试本地构建，也不要再额外探测这些命令。
2. 这种情况下，发布门禁里要把“本地构建未执行”单独写明，原因写清是“用户禁用”还是“环境缺失”，再继续完成其余文本、元数据、聚合和页面逻辑检查。
3. 只有用户明确允许，且本地环境具备时，才执行 `bundle exec jekyll build`。
4. 发布前必须做一次时间检视：除非用户明确要求按当天发布，否则不要把当前系统日期直接写成文章 `date` / `updated` / 文件名前缀。若只是因为“今天正好在改这篇”就用了当天日期，默认按门禁失败处理。

## 阶段划分

### 1. 选题定义

检查这些事：

1. 问题是什么。
2. 目标读者是谁。
3. 适用边界在哪里。
4. 这篇文章准备带来什么收益。

如果这一步都不清楚，后面的提纲和正文通常会发飘。

### 2. 提纲评审

检查这些事：

1. 一级标题有没有形成推进线。
2. 关键证据、例子、图示要放在哪里。
3. 是否有不该展开的支线。

结构问题优先转给 `$blog-outline-doctor`。

### 3. 初稿产出

检查这些事：

1. 有没有先把完整意思写出来。
2. 有没有过早进入修辞打磨。
3. 有没有因为追求“像文章”把真实判断写丢。

表达问题优先转给 `$blog-writing-style`。

### 4. 技术校验

检查这些事：

1. 术语是否准确。
2. 链路是否说得通。
3. 代码片段、图示、结论与反例是否自洽。
4. 是否存在明显事实风险。

事实风险优先转给 `$blog-fact-risk-review`。

### 5. 可读性精修

检查这些事：

1. 标题层级是否清楚。
2. 段落节奏是否自然。
3. 摘要是否能独立成立。
4. 结尾有没有和正文承诺脱节。

### 6. SEO 与站点分发检查

检查这些事：

1. `title`、`description`、slug、标签是否一致。
2. `series`、`featured`、`hot` 等字段是否自洽。
3. 相关文章、标签页、归档页、首页聚合位是否能正确收录。
4. SEO 入口是否破坏视觉系统约束。

SEO 和聚合问题优先转给 `$blog-seo-governor`。

### 7. 发布前门禁

这一步只做最后确认，不替代前面的阶段。

## 发布门禁

### 1. 内容与渲染门禁

1. `bundle exec jekyll build` 构建通过。
2. 关键页面渲染正常：`/`、`/archive/`、`/tags/`、`/about/`。
3. 新文 permalink、标签锚点、目录锚点可跳转。
4. 首页精选、热门、专题卡片无空字段、无死链接。
5. 文件名前缀日期、frontmatter `date`、`updated` 与实际时间编排一致；若用户没有明确要求“当天发布”，不得把系统当天日期当默认发布日期。

补充例外口径：

1. 如果用户明确禁止本地构建，或当前环境没有 Ruby / Bundler / Jekyll，这一项不能伪装成已验证，也不能擅自补跑。
2. 此时只能标记为“本地未执行”，并把最终构建门禁下沉到 CI / 线上构建结果。

### 2. 质量门禁

1. 开头 200 字内说清问题和重要性。
2. 正文兑现开头承诺，不要前后脱节。
3. 文章类型与质量尺子匹配，不拿单一模板硬套所有文章。
4. 没有明显空话、假深度、脑补结论。
5. 至少过“通用三问”：核心问题明确、判断依据具体、不确定性或边界写出。
6. 不要向目标读者传递明显错误的职业信号，例如把行业基线写成主发现，或把阶段性补课写成高阶洞见。

补充口径：

1. “开头 200 字内说清问题和重要性”不等于“必须结论先行”。
2. 如果文章的价值在于把答案一步步推出来，默认允许先立问题、场景和张力，再在正文中推出结论。

类型判断优先转给 `$blog-article-fit-checker`。  
总检优先转给 `$blog-quality-gate`。

### 3. 元数据与分发门禁

1. Frontmatter 字段完整且格式合法。
2. 无明显硬编码运营数据，例如虚构访问量、伪造统计。
3. SEO、标签、专题、相关文章、首页聚合规则通过。
4. `draft`、`featured`、`hot`、`series_buildable` 等状态不打架。

### 4. 发布确认门禁

1. 推送后 GitHub Actions 的 Pages 工作流成功。
2. 线上关键入口可访问。
3. 新文真实出现在预期聚合位。

## 降级发布规则

如果文章还没达到当前档位的及格线，不一定只能继续压着不发。  
可以降级处理，但要显式标注。

可接受的降级标注：

1. `草稿`
2. `阶段性记录`

适用场景：

1. 核心观点成立，但证据还不够完整。
2. 判断已有价值，但长期结果还没出来。
3. 结构和表达已可读，但还不适合当成完整复盘发布。

## 页面核验清单

### 1. 文章页

1. 标题、摘要、时间显示正常。
2. 目录锚点正常。
3. 文内跳转和相关文章链接正常。
4. 标签跳转正常。

### 2. 首页

1. 精选位无空字段。
2. 热门位排序和数据正常。
3. 专题位聚合正常。
4. 状态条指标真实且可核查。

### 3. 标签页与归档页

1. 新文已进入对应分组。
2. 锚点可跳转。
3. 列表项时间、摘要、标签显示正常。

## 输出要求

使用这个 skill 时，默认输出四部分：

1. 当前阶段：
   - 文章目前停在哪一步
   - 下一步该做什么
2. 未过门禁：
   - 哪些门禁还没过
   - 为什么还不能发
   - 若本地构建未执行，必须显式写出未执行原因，不能只写“环境问题”
3. 页面与分发检查：
   - 哪些页面和聚合位需要确认
   - 哪些入口可能有问题
4. 发布结论：
   - 可发布 / 接近可发布 / 还不能发

## 红线

1. 不要跳过前置阶段直接给发布结论。
2. 不要把发布门禁当成正文润色器。
3. 不要为了“快发”跳过构建、页面核验和聚合检查。
4. 不要把 SEO 曝光建立在虚构数据或破坏视觉系统约束之上。
5. 不要因为题材敏感就一刀切地不发，更要判断风险来自题材本身，还是来自错误 framing、边界缺失和成熟度信号暴露。
