# 文章质量门禁

## 核心定位

这个 skill 是总检，不是细修。

它回答的是这几个问题：

1. 这篇文章现在能不能过线。
2. 最大的问题在结构、文风、AI 味、事实风险，还是信息密度。
3. 先改哪里最划算。
4. 应该交给哪个已有 skill 继续处理。

它默认不直接大改全文。  
它更像一个门禁和分诊器。

先补一条总原则：

1. 不要拿一套模板判断所有文章。
2. 总检前先判断文章类型。
3. 文章类型错了，后面的质量结论通常也会一起错。
4. 技术内核扎实，不等于文稿已经过线。总检时必须把“内容判断是否成立”和“文章读起来是否像真实写作者完成”拆开判断。

默认先用 `$blog-article-fit-checker` 判断这篇是方案文、判断文、拆解文、观察文，还是过程文。  
这个 skill 不再默认要求所有文章都给出落地步骤、结果指标、决策框架或完整闭环。

## 检查维度

### 1. 开头是否立住

检查这些事有没有在前 200 字内说清：

1. 当时系统处在什么阶段。
2. 哪些前提已经定了。
3. 现在只讨论什么。
4. 真正卡住什么。
5. 如果文章背景涉及本地化交付转 SaaS、项目制转产品化、私有化转共享化等阶段变化，是否把“旧形态是什么”“当前最优先验证什么”说清了。

如果这是独立文章，开头必须能独立成立。  
如果这是承接上文的文章，就必须写清承接关系。

再补一个硬检查：

1. 正文里是否还残留 `TODO`、`待补`、作者口语批注、转写碎句或“这里没说完/需要优化”这类未完成标记。
2. 这类内容默认按“草稿未完工”处理，而不是普通措辞问题。
3. 正文里是否还残留作者明确批注的 `事实性错误`、`AI痕迹重`、`很怪`、`不像人写`、`这里不对`、`按评论修改` 等评论性标记。
4. 这类内容默认按“当前文件仍有未处理批注”处理，优先级高于普通润色建议。

### 2. 标题和大纲是否有逻辑

重点看：

1. 二级标题是不是一条线往前推。
2. 三级标题是不是服务对应二级标题。
3. 开头承诺和后文展开是不是对得上。
4. 有没有中途飞出新节点，导致拼凑感。

### 3. 文风是否自然

重点看：

1. 有没有总结腔、汇报腔、演讲腔。
2. 有没有说教味、布道味、营销味。
3. 有没有第二人称和强行共情。
4. 有没有元说明腔和方法论广播。
5. 有没有把普通技术判断不断抬高成立意句，导致一段比一段更像“上价值”。
6. 有没有语言过于凝练、句子过于紧绷的问题：作者觉得已经说清了，读者却需要自己补前提、补对象、补因果。
7. 开头有没有先解释作者准备怎么写，例如“先把背景讲清”“这篇只讲一件事”，而没有直接进入事实。
8. 标题有没有像作者自我旁白，例如“这篇最后该留下什么”“后来我最后怎么定下来的”，而不是在回答问题本身。
9. 某些关键小节是不是过于工整，像按同一个模板展开，导致读起来只有论述没有现场。

### 4. 技术内容是否扎实

重点看：

1. 有没有空技术细节。
2. 有没有泛公式、泛框架冒充深度。
3. 有没有具体对象、具体状态、具体风险。
4. 当前文章类型真正需要的关键信息有没有交代到位。
5. 给出的方案是否依赖不现实前提，例如默认总能找到 owner、默认组合交互能被单维度视角吸收、默认灰度适用于所有交付场景。
6. 给出的方案会不会把技术问题翻译成更多流程动作和沟通成本，例如拉表、认领 owner、开会确认，但没有带来自动化回归或可执行保护。
7. 如果文章在讲监控、指标、平台治理或其他机制建设，是否把“业务如何真正使用它”写清了；如果只写平台供给，不写业务侧摩擦，默认按没写透处理。
8. 如果正文引入了一个新系统、新机制或新角色，是否至少解释了它最小怎么工作；如果连基本工作流都没交代，默认按没写透处理。
9. 如果文章已经提到方案评审、争论、试运行、迁移决策等现场，是否真的把这些现场写出来了；如果只剩事后抽象总结，默认按“内容判断成立但文稿仍然发空”处理。
10. 如果文章写的是一套待做方案或差点落地的方案，是否把评审里真正被挑战的问题写出来了；如果方案没有被“打靶”，默认按没写透处理。
11. 如果文章真正卡住的是“这件事该不该做”，正文却顺手滑进低层实现比较，例如 `etcd/zk`、`watch/轮询`，默认按主线跑偏处理。
12. 如果作者本身是程序员、文章也在谈工程问题，但正文几乎只有原则判断和抽象 trade-off，没有 1 到 3 个能落到系统对象、运行机制或最小例子的技术抓手，默认按“理论多于技术细节”处理。

### 5. 事实风险是否可控

重点看：

1. 有没有伪造经历或脑补案例。
2. 有没有把推测写成事实。
3. 有没有时间、规模、组织关系写得过满。
4. 有没有明显需要证据但没给证据的结论。

### 6. 整体信息密度是否过线

重点看：

1. 每段是不是都在提供新信息。
2. 有没有重复包装同一个意思。
3. 有没有为了去 AI 味把真内容一起删掉。
4. 重点段落有没有展开到位。
5. 有没有把作者原本留作“待补内容”的标记直接删掉，却没有把那一段真正写完。
6. 有没有把本来应当很具体的业务现场，改成一篇干干的流程说明；如果全文都在讲“应该怎么做”，却看不见业务对象、使用动作和低摩擦路径，默认不能算过线。
7. 有没有“内容都对，但压得过头”的问题：段落像高密度摘要，信息很满，但普通读者跟不上文章在怎么推。

再额外看技术密度是否失衡：

1. 技术密度不足：有结论、有术语，但没有足够对象、状态、动作和代价去支撑判断。
2. 技术密度过高：术语和机制很多，但主线被淹没，读者抓不住文章到底在回答什么。
3. 技术密度错位：简单部分写太多，真正关键的技术判断写太少。
4. 技术密度失衡时，先判断它属于写透度问题、文风组织问题，还是两者都有。
5. 如果问题主要不是信息少，而是信息被压得太紧，默认按“写透度 + 文风”双问题处理，不要只给“风格偏硬”这种轻描淡写结论。

### 7. 文章类型和质量标准是否匹配

重点看：

1. 这篇文章真正属于哪一类。
2. 现在是不是在被错误模板要求。
3. 哪些质量项适用，哪些不适用。
4. 有没有因为错配标准把文章写假、写空、写成套路。

### 8. 职业信号风险是否可控

这部分不看“有没有观点”，而看文章会不会向同行、面试官、潜在合作方传递错误能力画像。

重点看：

1. 文章会不会让读者误以为作者缺乏基本风险意识、系统观或工程优先级判断。
2. 文章会不会因为 framing 错位，把局部问题写成作者唯一关注点。
3. 文章会不会因为前提、边界、代价没写清，让读者倒推出作者只看到了机制，没有看到系统。
4. 文章会不会把阶段性补课写成主洞见，反而暴露能力短板。
5. 文章会不会因为方案前提过于理想化，让读者倒推出作者没有真正面对 ToB 里的历史包袱、离职断层、私有化交付边界和组合爆炸。
6. 文章会不会把治理动作写成主要抓手，反而让读者觉得作者更擅长加流程，而不是减少系统脆弱性。

再单独检查一类高频风险：

1. 行业基线外显风险：像监控、告警、重试、幂等、隔离、死信、回放、限流、降级、权限、审计这类接近成熟工程团队默认基线的东西，是否被写成了“新的重大认识”。
2. 如果这类内容必须写，是否被准确降级成前提、背景、欠账、阶段性缺口或边界声明，而不是全文主亮点。
3. 真正值得展开的难点，是否落在基线之上的结构权衡，而不是停在“原来这里要做这个”。

## 门禁结论

默认只给三档：

### 1. 可发布

可以进入发布前检查。  
允许还有细小优化点，但主线、事实风险、文风、逻辑都已经过线。

### 2. 接近可发布

主线已经立住，但还有 1 到 3 个关键问题。  
修完这些问题才建议继续往发布走。

### 3. 明显不过线

这篇现在还停留在草稿阶段。  
通常是开头没立住、标题线断了、AI 味太重、事实风险明显、正文还留着未完成标记，或者几类问题一起存在。

## 输出要求

使用这个 skill 时，默认输出五部分：

1. 总评：
   - 可发布 / 接近可发布 / 明显不过线
2. 文章类型：
   - 主类型是什么
   - 当前判断是否错配
3. 内容质量与文稿质量拆分：
   - 技术内核是否成立
   - 文稿表达是否过线
   - 不要让“内容是对的”覆盖掉“写法仍然像生成稿”的问题
4. 致命问题：
   - 不解决就不能过线的问题
5. 重要问题：
   - 会明显拉低质量，但不一定致命
6. 修复路径：
   - 类型错配 -> `$blog-article-fit-checker`
   - 结构问题 -> `$blog-outline-doctor`
   - 内容没写透 -> `$blog-depth-reviewer`
   - 理论多于技术细节 / 技术抓手不足 -> `$blog-technical-grounding`
   - 文风问题 -> `$blog-writing-style`
   - 技术密度不足 / 错位 -> `$blog-depth-reviewer`
   - 技术密度过高 / 术语堆叠 -> `$blog-writing-style`
   - 语言过于凝练 / 高密度摘要腔 -> `$blog-depth-reviewer` + `$blog-writing-style`
   - AI 味问题 -> `$blog-ai-smell-checker`
   - 事实问题 -> `$blog-fact-risk-review`
   - 职业信号风险 / 行业基线外显风险 -> 先判断是否属于类型错配、framing 错位或边界声明缺失，再转 `$blog-article-fit-checker`、`$blog-outline-doctor` 或 `$blog-writing-style`

使用这个 skill 结束前，再补一个阻断动作：

1. 对当前文章运行一次未完成标记检查。
2. 至少检查：`TODO`、`待补`、`需要优化`、`没说清楚`、作者口语批注。
3. 只要命中任意一条，默认不能给“可发布”或“接近可发布”，必须先修掉或明确标成未完工。
4. 如果当前文章里存在作者评论、批注或显式纠错，再做一次“评论清零检查”。
5. 至少检查：`事实性错误`、`AI痕迹重`、`很怪`、`不像人写`、`这里不对`、`按评论修改`。
6. 只要这些评论还留在正文里且未被明确处理，默认不能给“可发布”或“接近可发布”。

## 红线

1. 不要只给一句“整体还行”。
2. 不要只讲优点，不讲不过线的问题。
3. 不要把所有问题混成一类，要指出问题属于哪一维。
4. 不要越权替代细分 skill 做整篇深改，这个 skill 的职责是判断和分诊。
5. 不要因为某个题材可能暴露风险，就简单给出“不要写”。
6. 真正要判断的通常不是题材能不能写，而是它是不是被写成了错误的主亮点、错误的成熟度信号，或错误的能力画像。
7. 不要因为文章技术判断扎实，就放松对 AI 味、收束方式和表达节奏的门禁。
8. 不要因为方案方向大体正确，就放过它所依赖的现实前提；理想化前提没有被写出来，同样会让文章不过线。
9. 不要因为文章已经有一套看起来完整的方案，就放松对“做事感”的门禁。没有评审挑战、没有讨论张力、没有被追问的关键点，方案文同样可能不过线。
