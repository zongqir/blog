文章类型匹配与质量尺子

核心定位

这个 skill 不回答“这篇是不是标准技术博客”。

它先回答三件事：

  1. 这篇文章本质上是哪一类。
  2. 现在是不是在被错误的质量标准要求。
  3. 这篇文章该用哪把尺子，而不是被哪套模板硬套。

它的职责不是把文章修成“方案文”或者“复盘文”。
它的职责是阻止错误模板把文章写坏。

补充参考：

  1. references/write-principles.md 归档了原始 write.md。
  2. 其中最重要的保留口径是：先有突出亮点，再谈完整性。
  3. 这个参考文件不是第二套总章程，而是给当前 skill 和相关 blog skill 提供共同参考。

硬底线

不分文章类型，先过三条最低要求：

  1. 这篇文章在说什么问题或观点。
  2. 作者的判断依据是什么。
  3. 作者目前不确定什么，或者边界在哪里。

如果这三条都答不上来，文章通常还不适合继续往发布走。

亮点优先

不要要求一篇文章面面俱到。
先判断这篇文章最值得读者带走的一个点是什么。

亮点通常落在这几类里：

  1. 一个高价值决策。
  2. 一个关键踩坑。
  3. 一个可复用框架。
  4. 一个反直觉结论。
  5. 一个可验证结果。

如果一篇文章主亮点已经成立，就不要为了追求“完整模板”把它写散。

再补一条亮点边界：

  1. 不要把接近行业公认基线的东西误写成主亮点。
  2. 像监控、告警、重试、幂等、死信、隔离、限流、降级这类公共卫生项，默认更适合写成前提、背景、欠账或边界，而不是“新的重大认识”。
  3. 真正值得成为文章亮点的，通常是基线之上的判断、权衡、结构代价和系统边界。

瑕疵声明

允许文章不完整，但必须把不完整写出来。
可以有瑕疵，不能装作没有瑕疵。

常见可接受表达：

  1. 这是阶段性判断，还缺长期数据。
  2. 这里主要是个人偏好，不是团队标准。
  3. 这部分尚未验证，先记录观察点。

默认文章类型

先判断主类型，再看是否有次类型。
不要一上来把一篇文章判成“全都要”。

1. 方案 / 复盘文

适用信号：

  1. 文章在回答“当时怎么选”“最后怎么做”。
  2. 正文会展开方案、权衡、实现、结果。
  3. 读者期待拿到可复用的做法或迁移顺序。

这类文章适合看：

  1. 决策点。
  2. trade-off。
  3. 落地路径。
  4. 验证方式。

2. 判断文

适用信号：

  1. 文章重点是把一个问题看清、说准。
  2. 文章核心价值是判断，而不是完整方案。
  3. 作者当时未必已经有完整落地路径。

这类文章适合看：

  1. 问题有没有说准。
  2. 前提和边界是否交代清楚。
  3. 判断依据是否成立。
  4. 有没有把不知道的部分硬补成结论。

不默认要求：

  1. 完整落地方案。
  2. 迁移步骤。
  3. 结果指标。

3. 拆解文

适用信号：

  1. 文章重点在拆对象、拆层次、拆矛盾。
  2. 文章价值来自把复杂问题看清，而不是给最终方案。
  3. 正文往往围绕对象、边界、职责、故障面推进。

这类文章适合看：

  1. 拆分维度是否稳定。
  2. 各层关系是否清楚。
  3. 抽象词有没有压回具体对象。
  4. 有没有把并列信息误写成主线节点。

不默认要求：

  1. 最终实施方案。
  2. 前后指标对比。
  3. 统一决策框架。

4. 观察文

适用信号：

  1. 文章重点是描述现象、识别模式、指出系统状态。
  2. 文章主要价值是让读者少误判，而不是给出完整打法。
  3. 内容更像“我后来越来越确定什么”，不是“我已经完整做完什么”。

这类文章适合看：

  1. 观察对象是否具体。
  2. 现象和判断之间有没有跳步。
  3. 有没有把局部经验写成普遍规律。
  4. 有没有拿空洞感悟冒充洞见。

不默认要求：

  1. 完整闭环结果。
  2. 统一操作手册。
  3. 每节都落到执行步骤。

5. 过程文

适用信号：

  1. 文章重点是记录当时怎么犹豫、怎么分叉、怎么慢慢想清楚。
  2. 价值来自真实过程，而不是最后整理出的标准答案。
  3. 文中允许保留未闭合的问题。

这类文章适合看：

  1. 过程是否真实。
  2. 分叉与犹豫是否有信息量。
  3. 有没有为了显得完整把过程写成结果。
  4. 有没有把后来知道的结论强塞回当时视角。

不默认要求：

  1. 全部问题都收束成结论。
  2. 一定有最佳实践。
  3. 一定有复用模板。

判断顺序

按这个顺序走，不要跳：

  1. 先看开头承诺的是什么。
  2. 再看正文主要在提供什么价值。
  3. 再看读者真正会从这篇里拿走什么。
  4. 最后再判断哪些质量项适用，哪些不适用。

如果一篇文章主承诺是“看清问题”，就不要拿“为什么不给完整落地方案”判它不过线。
如果一篇文章主承诺是“给出方案复盘”，那就不能只停在抽象判断。

常见误判

1. 把所有文章都按方案文判断

这是最常见的问题。

表现通常是：

  1. 强行要求落地步骤。
  2. 强行要求结果指标。
  3. 强行要求统一决策框架。
  4. 强行要求“为什么选 A 不选 B”必须完整闭环。

这会把判断文、拆解文、观察文、过程文全部写假。

2. 把“没有完整方案”误判成“质量差”

有些文章的价值就在于把问题看准。
当时没有完整方案，不是缺陷；硬补一套方案才是缺陷。

3. 把“没有指标”误判成“没有价值”

不是每篇文章都在讲结果。
有些文章写的是系统状态、问题拆解、边界识别，这些不一定天然带前后指标。

4. 把“没有框架”误判成“结构不清”

结构清楚，不等于必须有方法论框架。
很多文章只要对象、边界、推进线清楚，就已经成立。

5. 把行业基线误判成文章洞见

这也是高频误判。

表现通常是：

  1. 把成熟团队默认会关注的公共卫生项写成主发现。
  2. 把“以前没做到，现在补上了”写成高阶判断，而不是阶段性欠账。
  3. 文章停留在“原来这个要做”，没有继续展开更高一层的系统问题。

这会让文章的亮点错位，也可能向读者暴露错误成熟度信号。

低质量信号

这部分不是通用模板，而是跨类型都成立的坏信号。

通用低质量信号

  1. 文章承诺和正文实际提供的内容不一致。
  2. 抽象词很多，但具体对象、动作、状态很少。
  3. 为了显得完整，补了作者当时并没有的方案、指标或结论。
  4. 用方法论口气盖过真实内容，看起来像在发表正确意见。
  5. 把行业基线写成主洞见，导致文章亮点和作者能力画像一起错位。

类型错配信号

  1. 判断文被写成假方案文。
  2. 拆解文被写成观点拼盘。
  3. 观察文被写成空洞感悟。
  4. 过程文被事后整理成过度工整的标准答案。
  5. 方案文却没有真正交代决策点、代价和落地对象。

类型化自检

不是每篇都过同一张清单。
先判断类型，再用对应自检。

1. 方案 / 复盘文自检

  1. 有没有把决策点说清。
  2. 有没有交代为什么这样选。
  3. 有没有写清对象、代价和边界。
  4. 有没有给出落地路径或验证方式。

2. 判断文自检

  1. 问题有没有说准。
  2. 判断依据是否可理解。
  3. 有没有把不知道的部分假装知道。
  4. 读完后读者能不能明白“为什么会这样判断”。
  5. 有没有把行业基线本身写成主发现，而没有继续推进到真正值得判断的那一层。

3. 拆解文自检

  1. 拆分维度是否稳定。
  2. 对象和边界是否清楚。
  3. 各层关系是否能被读者跟住。
  4. 有没有用空概念代替真实系统状态。
  5. 有没有把公共卫生项写得煞有其事，反而压住了更关键的结构关系。

4. 观察文自检

  1. 观察对象是否具体。
  2. 现象和判断之间是否有足够支撑。
  3. 有没有拿局部现象硬推普遍规律。
  4. 有没有把感受写成分析。

5. 过程文自检

  1. 过程是否真实而不是事后排演。
  2. 犹豫和分叉是否带来新信息。
  3. 有没有为了收尾把未解问题硬写成已解决。
  4. 读者能不能理解作者当时为什么会那样想。

发布强度档位

除了“文章类型”，还可以补看“发布强度档位”。
这不是第二套类型系统，而是一个更粗的门槛辅助。

1. 决策复盘档

适用：架构选型、线上故障复盘、重大技术取舍。

建议重点看：

  1. 开头是否说清问题、影响、成功标准。
  2. 有没有至少一个明确决策及依据。
  3. 有没有 trade-off、失败路径或回滚点。
  4. 有没有验证方式。

2. 实践笔记档

适用：工具实践、迁移过程、阶段性总结。

建议重点看：

  1. 场景和约束是否清楚。
  2. 有没有核心步骤或关键片段。
  3. 有没有至少一个坑和处理方式。
  4. 有没有不建议照搬的场景。

3. 个人感想档

适用：职业反思、方法论感受、学习心得。

建议重点看：

  1. 是否明确是个人视角。
  2. 有没有至少一个具体经历或触发事件。
  3. 有没有说明这条感想不成立的场景。

输出要求

使用这个 skill 时，默认输出四部分：

  1. 文章主类型：
    • 主类型是什么。
    • 有没有次类型。
  2. 当前错配：
    • 现在被什么错误标准要求了。
    • 这些要求为什么会把文章写坏。
  3. 适用质量尺子：
    • 这类文章该检查什么。
    • 不该强求什么。
  4. 修正建议：
    • 哪些通用模板要求要拿掉。
    • 接下来更适合用哪个 blog skill。
    • 如存在行业基线外显风险，要明确指出问题出在亮点选错、framing 错位，还是边界声明不足。

红线

  1. 不要拿一套模板判断所有文章。
  2. 不要把“没有方案”自动判成质量差。
  3. 不要把“没有指标”自动判成没价值。
  4. 不要为了过门禁逼作者补自己当时没有的内容。
  5. 不要把文章修成比真实过程更工整的假成品。
  6. 不要把行业基线默认当成文章亮点，除非文章真正讨论的是这些基线为什么在特定场景下会失效、冲突或长歪。

持续记录那些真正影响结果的工程取舍。
