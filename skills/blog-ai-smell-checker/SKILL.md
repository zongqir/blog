# 博客 AI 味诊断

## 核心定位

这个 skill 不负责判断“这篇是不是 AI 写的”。

它只做一件更可靠的事：

找出文章里**为什么会让人觉得像 AI 生成**。

也就是说，它默认输出的是：

1. 哪些地方有 AI 味。
2. 这些 AI 味具体体现在哪。
3. 为什么读起来会像模型在生成。
4. 先改哪里最有效。

它不输出：

1. `这是 AI 写的。`
2. `这不是 AI 写的。`
3. 没有证据支撑的真伪判断。

## 这个 skill 看什么

重点查下面这几类高风险生成痕迹：

### 1. 拼凑感

典型症状：

1. 开头承诺了一套结构，正文却写成了另一套。
2. 中途不断飞出新节点，像写着写着想到什么就补什么。
3. 标题之间没有明显关系，只是相关知识点并排摆着。
4. 先聊抽象技术主题，后面再零散补业务理由和落点。

### 2. 句子和段落过度平均

典型症状：

1. 每句都差不多长。
2. 每段都差不多满。
3. 节奏太匀，没有轻重。
4. 每段结尾都像一个小总结。

这种文章通常看起来很工整，但不像真实写作者。  
真实技术文章经常会在局部停住，不会每段都收得很干净。

### 3. 过度完整

典型症状：

1. 背景、定义、意义、方案、总结都齐。
2. 文章像优等生标准答案，几乎没有停顿和留白。
3. 原文本来只想讲一个局部判断，最后被补成一套全景方法论。
4. 作者没定义术语，文章却自动补了一整套概念说明。

过度完整不一定更好。  
它经常会把真实复盘里本来成立的局部判断，修成一篇没有现场感的完整作文。

### 4. 空心技术细节

典型症状：

1. 技术细节看起来正确，但没有信息量。
2. 用泛公式、泛估算、泛框架假装深度。
3. 只写“资源压力、状态治理、复杂度上升”这种大词，不写具体连接、队列、cursor、backlog、恢复过程。
4. 组件名很多，但没有写对象关系、状态变化和代价出现的方式。

技术真实感不等于术语密度。  
更重要的是写清：

1. 哪个状态多起来了。
2. 哪个恢复动作开始变慢了。
3. 哪类故障开始从点状变片状。
4. 哪种配置开始越来越难改。

### 5. 假洞察

典型症状：

1. 句子看起来高级，但谁都能说。
2. 只是把原文换成一个更大的抽象词，再说一遍。
3. 用“这背后反映的是”“更深层的问题在于”“归根结底，这是一个……问题”抬高语气。
4. 把常识层面的现象重新命名成更大的概念，比如把“环境变多、结果不一致”抬成“某个模型失效”“行为脱钩”，但没有增加新的事实含量。
5. 开头先解释写作动作，例如 `这篇得先把业务背景说清`、`先把前提摆一下`、`这篇先讲背景再讲判断`。这类句子通常不是事实内容，而是在给写作动作搭台子。

如果要上升抽象，必须立刻落回具体对象和动作。  
洞察不是把“负载问题”改写成“系统治理问题”，而是写清它为什么会在那个阶段变成治理问题。

### 6. 过度表演的人味

典型症状：

1. 大量使用 `其实`、`当然`、`通常`、`真正`、`这也是为什么`。
2. 标题和过渡句像一个疲惫老专家在总结，而不是在复盘事情。
3. 为了“像人”，刻意堆犹豫、转折、感慨。

### 7. 方法论腔和元说明腔

典型症状：

1. 正文里先讲“我要怎么分析这件事”。
2. 正文里解释“这篇怎么写”“这一段为什么先写”。
3. 把方法框架直接铺在读者脸上，像培训材料，不像博客。
4. 大量使用“这里想讨论的是”“这一部分主要说明”“接下来我们看”这种带路句。

再补一类高风险句型：

1. `这篇我其实只想讲……`
2. `如果压成一句话……`
3. `我更愿意把它理解成……`
4. `真正值得讨论的是……`
5. `如果要我把这件事压成一句话，我会更愿意这样说……`
6. `如果要把这篇文章压成一句话，我的答案是……`
7. `这篇得先把业务背景说清`

这类句子默认按高风险 AI 痕迹处理。  
它们的问题不在信息本身，而在于作者先退后一步解释“我要怎么总结”，像模型在给自己的结论搭台子。除非用户明确要求这种句式，否则默认直接改成结论本身，不保留客套前摇。

### 8. 判断太圆

典型症状：

1. 每个观点都兼顾多方、全面平衡。
2. 没有偏好、没有犹豫、没有不耐烦，也没有具体取舍。
3. 读完只觉得“都对”，但感受不到作者真实站位。

这种问题往往不是“客观”，而是把作者修没了。

### 9. 背景悬空

典型症状：

1. 开头没交代系统阶段、已定前提、当前讨论边界。
2. 看完前几段还不知道为什么会讨论这个问题。
3. 明明是独立文章，却默认读者知道上文。
4. 开头直接讨论性能、治理、架构、模型，但没有写是哪类业务把这个问题逼出来的。

### 10. 现场感不足

典型症状：

1. 有 topic、有 consumer、有 shard。
2. 术语都对，但没有现实触发点。
3. 看不出作者为什么会在那个点开始别扭、犹豫或转向。
4. 机制讲了很多，但回答不了“业务为什么在意这个”。

例子看起来正确，但没有现场感，也是典型 AI 痕迹之一。

再补一类常见来源：

1. 正文已经提到方案评审、争论、试运行、值班或迁移现场，但后面没有把这些场景展开。
2. 文章只剩“我后来意识到”“更本质的问题是”这种事后总结口气。
3. 读者读完知道作者得出了什么判断，却看不到这判断是在什么场合被逼出来的。
4. 明明在讲一套待做方案，正文却没有写方案是怎么被追问、被挑战、被一层层拆开的。

### 11. 为了讲机制而讲机制

典型症状：

1. 单开一节解释某个能力、机制或原理，但和全文主线没有真实关系。
2. 这一节删掉以后，文章反而更顺。
3. 技术点本身没错，但读者看不出它为什么在这里出现。

这类问题表面上像“内容不够聚焦”，本质上常常是主线没立住。  
文章不是因为技术点不对而像 AI，而是因为它在替结构空洞补材料。

### 12. 主抓手被机制带跑

典型症状：

1. 开头立的是业务问题，正文却逐渐变成某个机制的说明文。
2. 读者本来以为文章在回答“为什么会这样”，最后却主要读到“这个机制是什么”。
3. 文章嘴上在讲业务，笔墨却主要花在 batch、ack、DLQ、Router 这类单点机制上。
4. 文章真正停在“这件事值不值得做”，却顺手补出 `etcd/zk`、`watch/轮询` 这类并不存在的实现层比较。

这类问题会让文章读起来很怪。  
不是因为机制讲错了，而是因为题眼和正文重心已经分家了。

### 12.2 只有论述，没有做事感

典型症状：

1. 整段都在总结为什么重要、为什么别扭、为什么不该做。
2. 但读者看不到评审现场、挑战问题、讨论转向或谁在追问什么。
3. 文章像在写一份复盘结论，不像真的经历过那次方案讨论。

这种味道很重，因为模型最擅长把事情总结圆，却不擅长保留“人是怎么把这件事谈崩或谈透的”。

### 12.1 标题像作者在总结自己

典型症状：

1. 标题或小标题更像作者的自我旁白，而不是这一节的主题。
2. 例如 `这篇最后该留下什么`、`后来我最后怎么定下来的` 这类写法。
3. 标题没有告诉读者这一节在回答什么，只是在说明作者准备怎么收束。

这种问题很常见，因为模型会天然想给自己的总结动作命名。  
真实写作者更常直接写对象、判断或动作本身。

### 13. 过度升格和立意句过密

典型症状：

1. 文章反复使用“不是 X，而是 Y”去抬高立意。
2. 普通判断被一层层改写成更大的抽象判断。
3. 几乎每个小节都试图给出一个“真正值得讨论的是……”式的收束句。
4. 技术判断本来已经成立，文章却还在继续往“模型”“本质”“真正问题”上升。

这种问题的危险不在于句子错了。  
而在于作者像一直站在台上解释自己看得多深，呼吸感和现场节奏会被压平。

### 14. 结构过度工整

典型症状：

1. 各个小节篇幅接近，信息功能也接近，像按同一模板展开。
2. 文章稳定落在“现状 -> 痛点 -> 分析 -> 方案 -> 总结”这种标准周报骨架里，读起来太顺，反而不像真实写作者。
3. 每段都先抛判断、再列要点、最后收一句结论，节奏过于统一。

这类问题不一定会让内容变错。  
但会让文章像被格式先行地整理过，真实思路和呼吸感都被磨平。

### 15. 结尾注入痕迹或导流硬过渡

典型症状：

1. 正文已经自然结束，后面突然补出一段为了挂相关文章、专题或延伸阅读的过渡句。
2. 过渡句与上一段的对象关系不连续，像被额外塞进一个运营任务。
3. 句子形式高度相似，例如连续出现两句“如果继续往……看，也可以接着看……”。
4. 读起来不是作者真的想到下一步，而是在完成“顺手把这两篇也带出去”的额外指令。

这种痕迹可以来自多种来源：站点运营、写作提示、额外硬要求。  
诊断时不要武断声称是 prompt 注入，但要把它识别成真实语感已经被外部任务打断。

### 16. 金句式收束

典型症状：

1. 段尾或结尾突然压出一句短而重的判断，像在提炼“中心思想”。
2. 句子看起来有力，但和上文的对象、动作、边界没有完全接上。
3. 高频出现 `X，本身就是Y`、`问题问到这里，才算……` 这类短促判断句。

这种句子的问题通常不是“句子不对”。  
而是它太像为了制造洞见感而补出来的收束动作。

## 检查顺序

### 1. 先看开头是不是悬空

检查这四件事有没有出现：

1. 当时系统处在什么阶段。
2. 哪些前提已经定了。
3. 现在只讨论什么。
4. 真正卡住什么。

如果是技术分析文章，再加一条：

1. 业务触发点有没有出现。
2. 是什么业务结果、处理差异或失败代价把问题逼出来的。

如果这四件事缺了两件以上，整篇很容易从开头就带 AI 味。

### 2. 再看承诺和兑现是不是脱节

重点检查：

1. 开头列出来的几种方案，正文有没有真的按这个展开。
2. 有没有中途新长出来的重要节点。
3. 结尾是不是在总结一个开头没承诺过的方向。

这种脱节是“按块生成感”的核心来源之一。

### 3. 再看技术细节是不是空

优先抓这种句子：

1. 正确但空的公式。
2. 听起来专业但没有具体对象的抽象词。
3. 只写大风险，不写具体机制和观测面。
4. 只报组件名，不写对象关系、状态变化和代价。

### 4. 最后看语气是不是演过头了

重点检查：

1. 有没有过多总结腔。
2. 有没有过多“不是 A，而是 B”。
3. 有没有站在台上点评系统的口气。
4. 有没有刻意模拟经验感的副词堆叠。
5. 有没有靠连接词硬拽逻辑，而不是靠事实顺序和对象关系成立。
6. 有没有把普通判断包装成郑重大洞察。
7. 有没有自问自答式过渡，好像作者在一边写一边解释自己的写法。
8. 有没有在结尾突然切到抽象总结或站点导流，像另一篇文的开头。
9. 有没有技术是对的，但每段都在持续升格，像在反复证明自己看到了“更深一层”。
10. 有没有大量使用“第一/第二”“更现实的”“核心不是……而是……”这类强引导连接词，像在领读者按模板前进。
11. 有没有段尾反复补金句式判断，让文章每走几段就像要交一次总结。

少靠这些连接词制造逻辑感：

1. `因此`
2. `然而`
3. `与此同时`
4. `进一步而言`
5. `换言之`
6. `某种程度上`
7. `从另一个角度看`

## 输出要求

使用这个 skill 时，默认输出四部分：

1. 总体判断：
   - 这篇最重的 AI 味是哪一类。
   - 是不是缺业务触发点，导致整篇在悬空聊技术主题。
   - 技术内核是否成立，以及它为什么仍然不能抵消 AI 味判断。
2. 证据位：
   - 逐条指出原文中最像 AI 的地方。
   - 标出哪些段落是在为了讲机制而讲机制。
   - 标出哪里已经被单一机制带跑，和开头的题眼脱节。
   - 标出哪些结尾或过渡像在执行额外导流任务，而不是自然推进正文。
3. 原因说明：
   - 为什么这会让人觉得像 AI，而不是像真实复盘。
   - 若怀疑是外部硬要求打断语感，只能写成“像被额外任务插入”，不要把猜测直接写成事实。
4. 修改优先级：
   - 先改开头、先改标题线、先删空技术细节，还是先压语气。

如果适合继续改写，额外补一句：

1. 更适合转给 `$blog-writing-style`
2. 还是更适合先转给 `$blog-outline-doctor`

## 红线

1. 不要把“有 AI 味”写成“确定是 AI 生成”。
2. 不要只给抽象结论，必须给证据位。
3. 不要只说“更自然一点”，必须说清楚哪里假、哪里空、哪里拼。
4. 不要把所有问题都归到文风，结构和承诺兑现脱节往往更关键。
