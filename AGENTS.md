# Repository Guidelines（仓库指南）

## 项目结构与模块组织
本仓库是一个基于 Jekyll 的 GitHub Pages 博客。

- `_posts/`：博客文章，文件名格式为 `YYYY-MM-DD-slug.md`。
- `_layouts/`：页面布局模板（如 `default.html`、`post.html`）。
- `_includes/`：可复用的 Liquid 片段（如文章列表渲染）。
- `assets/css/`、`assets/js/`：主题样式与前端脚本。
- `.github/workflows/pages.yml`：构建并部署到 `gh-pages` 的工作流。
- `scripts/new-post.ps1`：PowerShell 发文脚手架。
- 根页面：`index.md`、`archive.md`、`tags.md`、`about.md`、`404.md`。

## 构建、测试与本地开发命令
在仓库根目录执行：

```bash
bundle install
bundle exec jekyll serve
bundle exec jekyll build
```

- `jekyll serve`：本地预览（`http://127.0.0.1:4000`）。
- `jekyll build`：生成生产构建到 `_site/`。

创建新文章（PowerShell）：

```powershell
.\scripts\new-post.ps1 -Title "我的文章" -Description "摘要" -Tags 架构设计,SaaS化解读
```

## 编码风格与命名规范
- Markdown、HTML、CSS、脚本统一使用 UTF-8。
- 保持与现有文件一致的缩进风格（HTML/CSS/Liquid 使用 2 空格）。
- 面向站点展示的文案优先中文，表达简洁直接。
- 文章 frontmatter 固定使用：`title`、`description`、`date`、`updated`、`tags`。

## 博客写作补充约束
- 当任务涉及博客选题、提纲、初稿、改写、润色、复盘、发布时，默认先判断该用哪个 blog skill，不再把详细执行规则长期堆在 `AGENTS.md` 里。
- 博客相关总原则只保留不伤文的底线：真实性、可发布性、承诺与正文一致、不要脑补、不要用单一模板硬套所有文章。
- 如与本文件其他规则冲突，优先级顺序为：
  1. 真实性与安全性
  2. 可发布性
  3. 对应 blog skill 的明确约束
  4. 本文件其余通用写作规则

## 博客协作统一执行口径
博客相关规则默认按 skill 分工执行：

- 结构、大纲、标题推进：`skills/blog-outline-doctor/SKILL.md`
- 内容写透度与关键段落补写：`skills/blog-depth-reviewer/SKILL.md`
- 技术抓手、方案细节与运行机制补强：`skills/blog-technical-grounding/SKILL.md`
- 文风、表达、删改边界：`skills/blog-writing-style/SKILL.md`
- AI 痕迹清理与人味校正：`humanizer-zh`
- 文章类型与适用质量尺子：`skills/blog-article-fit-checker/SKILL.md`
- 总检与问题分诊：`skills/blog-quality-gate/SKILL.md`
- SEO、frontmatter、标签、专题、首页聚合：`skills/blog-seo-governor/SKILL.md`
- 写作流程、发布门禁、页面核验：`skills/blog-release-gate/SKILL.md`

这里只保留三条总原则：

1. 目标不是生成标准博客，而是把作者真实想法整理成可发布的文章。
2. 先判断问题属于结构、写透度、技术抓手、表达、类型错配、事实风险、SEO 或发布门禁中的哪一类，再调用对应 skill，不要所有问题都堆到一个地方解决。
3. 当用户对文章给出新批评时，先修当前文章，再把可复用约束沉淀到对应 skill，而不是继续把 `AGENTS.md` 写成一份越来越长的博客总手册。

## 博客技能迭代主线
本仓库关于博客的长期目标，不只是持续发文，也包括持续训练出一套作者真正满意、能稳定复用的 blog skills。

执行约束：
- 每次博客任务除了服务当前文章，也要顺手判断这次问题是否暴露了 skill 体系缺口。
- 当用户对文章给出明确投诉、负反馈或“这段很 AI / 很怪 / 不像人写的”这类评价时，不能只停在修文；还要额外判断这是文章局部问题，还是协作系统问题。
- 协作系统问题至少按四类归因：`skill 覆盖缺口`、`skill 触发或串行错误`、`执行没有严格按 skill 落地`、`验收标准过松`。
- 若结论属于上面四类之一，且约束稳定可复用，必须回写到对应 skill 或 `AGENTS.md` 的触发规则；不要只在当次回复里解释。
- 凡是用户基于评论、批注、投诉、负反馈继续推进修改的任务，每次修完当前内容后，都必须额外做一次“skill 增量检查”：判断这次评论是否暴露了新的可复用约束、触发规则、验收标准或执行偏差。只要结论可复用，就必须当次回写到对应 skill 或 `AGENTS.md`，不能留到以后再说。
- 不要因为“技术内核是对的”就默认文稿过线。技术正确只说明内容可用，不说明表达、节奏、收束和发布质量已经过线。

## 博客任务技能显性触发机制
任何博客相关任务，先做分诊，再进入对应 skill；不要跳过分类直接开始写。

默认执行顺序：
1. 先判断任务属于结构、写透度、技术抓手、表达、类型判断、SEO、发布门禁中的哪一类。
2. 若用户一句话同时包含多类问题，先用 `blog-quality-gate` 做分诊，再按最小必要集合调用其他 skill。
3. 若任务意图不清、跨阶段过多、或无法稳定判断主问题，默认先触发 `blog-quality-gate`，不要凭感觉直接选单一 skill。
4. 若用户用口语、语音转写式表达、重复修正、半句插话或不完整句子提出博客修改意见，必须先把口语翻译成明确任务，再决定触发哪个 skill；不要因为表述松散就停在字面理解。
5. 若当前文章正文里扫到 `TODO`、`待补`、`需要优化`、`没说清楚`、作者口语批注或其他未完成标记，不管用户当句怎么说，默认都要叠加 `blog-depth-reviewer`；因为这说明正文本身还没写完。
6. 若用户说“按我的评论改”“按批注改”“按我写的那些评论优化”，或正文里已经存在显式评论、批注、事实纠错、AI 味批注，默认先把评论转成逐条待办清单，再开始改文；不能跳过评论直接做自由发挥式优化。
7. 若用户明确要求“把 A 文里的这些内容塞进 B 文”，优先级最高的是用户点名的具体抓手、段落和判断，不是模型自己觉得更高级的抽象层。没有显性落到 B 文里，任务就不算完成。

强触发映射：
- 出现“选题”“标题”“提纲”“结构”“重组段落”“开头怎么起”等请求，默认触发 `blog-outline-doctor`。除非用户明确缩小范围，否则“标题”默认同时包括文章 `title`、二级标题、三级标题和其他显式标题文案。
- 出现“内容太浅”“不够透”“论证不够”“补关键段落”“写深一点”等请求，默认触发 `blog-depth-reviewer`。
- 出现“太虚”“大道理太多”“多加一点技术细节”“多一点技术方案细节”“多一点实战抓手”“没有讲技术本身”“得有点东西”这类请求，默认触发 `blog-technical-grounding`。
- 出现“润色”“改写”“压缩”“展开”“更自然”“去 AI 味”“像人写的”等请求，默认触发 `blog-writing-style`。
- 出现“很 AI”“AI 味重”“不像人写的”“像总结稿”“像生成稿”“去 AI 痕迹”“去机器味”等请求，默认同时触发 `blog-writing-style` + `humanizer-zh`，不要只靠单一 skill 处理。
- 出现“这篇像不像博客”“该写成复盘还是方案”“类型对不对”“要不要写结果/框架/落地”等请求，默认触发 `blog-article-fit-checker`。
- 出现“总检”“找问题”“审一下能不能发”“先看有什么硬伤”等请求，默认触发 `blog-quality-gate`。
- 出现“标签”“frontmatter”“SEO”“专题”“首页聚合”“运营字段”“相关文章”等请求，默认触发 `blog-seo-governor`。
- 出现“准备发布”“发布前检查”“上线前核验”“页面核验”“发布门禁”等请求，默认触发 `blog-release-gate`。

执行口径：
- 每次处理博客任务时，必须先显式说明“本次任务分类”和“本次触发的 skill”。
- 若任务需要串行使用多个 blog skill，必须说明调用顺序与理由，不要把多个 skill 混成一个模糊动作。
- 若没有触发某个看起来可能相关的 skill，应简短说明为什么跳过，避免黑盒判断。
- 用户直接点名 skill 时，按用户指定执行；若与真实性、安全性或可发布性冲突，先指出冲突再调整。
- 新的博客问题优先修当前文章；只有当约束可复用且稳定时，才回写到对应 skill 或本文件。
- 若任务属于文章复盘、投诉归因、skill 反思，默认先给出“本次失误归因”，明确它是文章局部问题，还是 `skill 覆盖缺口 / 触发错误 / 执行偏航 / 验收过松`。
- 若用户已经明确指出某段“很 AI”“很怪”“像总结稿”“像被硬接了个链接过渡”，默认不能只回一句“会更自然些”；要把投诉翻译成具体证据位和可执行规则，再决定是否回写 skill。
- `humanizer-zh` 默认不是单独兜底器。凡是触发它时，都要先判断问题究竟属于表达生硬、结构失真、内容被压扁，还是纯 AI 痕迹清理；除非用户明确只做局部去味，否则默认与 `blog-writing-style` 串行，必要时再叠加 `blog-depth-reviewer`。
- 博客协作默认保守删改：凡是准备删除用户现有正文中的段落、小节、列表项或其他有实质信息的内容，必须先明确告诉用户“准备删什么、为什么删”；用户未明确同意前，默认只能改写、压缩、重排或标注，不得直接删除。
- 在博客草稿里遇到 `TODO`、`待补`、口语批注、作者自我提醒或显式“这里没写完/需要优化”之类标记时，默认先按“内容未完成”处理，而不是按普通脏文本处理。优先补完、改写成正文或与用户确认，不得直接忽略。
- 任何博客任务在准备结束前，必须对本次修改文件做一次未完成标记扫描。至少检查：`TODO`、`待补`、`需要优化`、`没说清楚`、作者口语批注。命中任意一条，都视为当前文件不能收工；必须补完、改写成正文，或明确向用户说明仍未完成。
- 若当前文件里已经有作者评论或批注，准备结束前还必须额外做一次“评论清零检查”：逐条确认每条评论是 `已修复`、`保留待确认`，还是 `明确拒绝并说明原因`。只要还有未回应评论，默认不能把任务说成完成。
- 若本次任务是按评论、批注或投诉继续修文，准备结束前还必须再做一次“skill 回写检查”：确认这轮评论是否已经沉淀为新的可复用约束；如果没有检查，默认不能把任务说成完成。
- 正文里若出现 `事实性错误`、`AI痕迹重`、`很怪`、`不像人写`、`这里不对` 这类作者强批注，默认按高优先级阻断处理。先修对应句段，再谈整体优化；不能把它们埋在后续泛改里一起带过。
## 决策优先级与冲突处理
当规则冲突或目标冲突时，按以下顺序决策（从高到低）：

1. 真实性与安全性：不得伪造数据、误导读者或引入高风险改动。
2. 可发布性：页面可构建、可渲染、关键路径不报错。
3. 内容可读性：信息结构清晰、结论可验证、读者可执行。
4. 一致性：遵循既有信息架构、文案口径、视觉 token。
5. 美观度：在不破坏前四项前提下优化视觉表现。

执行约束：
- 不以“视觉好看”为理由破坏内容真实性。
- 不以“快速上线”为理由跳过构建与发布门禁。
- 不以“炫技动画”为理由损害性能与可读性。

## 博客写作 SOP（从选题到发布）
- 写作流程、阶段判断、发布前顺序与门禁，统一按 `skills/blog-release-gate/SKILL.md` 执行。
- 博客任务的最小总要求只保留三条：
  1. 开头 200 字内说清问题和重要性。
  2. 正文兑现开头承诺，不要前后脱节。
  3. 不拿单一模板要求所有文章；是否需要方案、落地、结果、框架，按 `skills/blog-article-fit-checker/SKILL.md` 判断。

补充口径：
- “开头 200 字内说清问题和重要性”不等于“必须结论先行”。
- 只有当文章类型、作者意图和阅读体验都适合时，才允许一上来先端最终判断。
- 如果文章的价值在于把答案一步步推出来，例如过程文、观察文、部分判断文，默认允许先立问题、场景、分叉和张力，再在正文里把结论推出来。

## 博客时间与文件名规则
- 对存货文，`date` 默认按内容编排顺序确定，不强求等于真实写作时间。
- 若素材跨多年、真实时间已不精确，优先保证文章之间的相对顺序、系列顺序和问题演进顺序正确，不要伪造精确到分钟的真实时间。
- 若用户已经明确表示“你自己定一个合理时间就行”“我不想为时间编排再做决策”，默认由协作者直接给出一套够自然的编辑时间分布并执行，不要再把日期选择题抛回给用户。
- 除非用户明确要求“按今天发布”“就用当天日期”或当前任务本身就是当天即时发布，否则禁止把当前系统日期直接当作草稿或 `_posts/` 文章的默认 `date`、`updated`、文件名前缀日期。为了省事直接写成当天日期，默认按时间编排错误处理。
- 如果用户已经明确表达过“不按今天来”“不要用当天日期”“不要按当前时间自动填”，这条约束视为高优先级长期偏好，后续博客任务默认继承，不得再次用当前日期试错。
- 对 `_posts/` 下的文章，文件名前缀日期必须与 frontmatter 的 `date` 保持一致；发现不一致时，默认按一致性错误处理，不要自由保留。
- `updated` 默认与 `date` 保持一致。只有当用户明确要求保留“后续修订时间”或当前任务本身就在维护真实修订时间时，才允许单独设置不同的 `updated`。
- 若同一篇文章出现“文件名前缀日期 / date / updated”三者不一致，默认先收敛到 `文件名前缀日期 = date = updated`，再根据用户明确要求做例外。
- 每次准备发布博客时，默认做一次“时间检视”：
  1. 这篇内容应排在前后哪些文章之间。
  2. 文件名前缀日期与 frontmatter `date` 是否一致。
  3. `updated` 是否默认与 `date` 一致，若不一致是否有明确理由。
  4. 是否会让读者误以为这些文章是在极短时间内集中写成。
  5. 若属于系列文章，时间顺序是否符合内容依赖关系。
  6. 当前日期如果是 `2026-03-15` 这类系统当天，是否被误用成文章发布日期；若没有明确“当天发布”意图，必须改掉。
- `_draft/` 默认允许并优先使用中文文件名，便于作者识别和管理素材。
- `_posts/` 允许使用中文文件名。若发布文章文件名使用中文，默认必须显式设置 `permalink`，不要再让 URL 直接跟随中文文件名。
- 中文文件名的发布文章，`permalink` 默认使用拼音风格，由协作者自动生成；不要要求作者自己手动想英文 slug。
- 拼音 `permalink` 的目标是稳定、可复制、不容易出错；默认不追求翻译感，也不追求营销化命名，只要能对应标题主语义即可。
- 若用户没有明确要求其他风格，发布文章的 `permalink` 默认优先级为：
  1. 拼音风格
  2. 英文概括型
  3. 中文 URL
- 若文章已经发布并对外传播，修改文件名或 `permalink` 前必须先提醒用户这会影响旧链接；未经确认，不要擅自改已发布文章 URL。

## 测试与验证指南
本仓库无单元测试框架，采用按风险触发的构建与页面核验：

1. 若改动涉及 `_layouts/`、`_includes/`、`assets/`、根页面、配置、脚本、CI 或其他可能影响渲染/构建的文件，本地执行 `bundle exec jekyll build`，确保构建通过。
2. 若只是文章正文、frontmatter、标签、摘要、措辞等内容修订，默认不要求每次都跑 `jekyll build`；优先做文本核对，只有在用户明确要求发布前核验、任务进入发布门禁阶段，或你判断页面渲染风险已经上升时再构建。
3. 抽查关键页面：`/`、`/archive/`、`/tags/`、`/about/`。
4. 更完整的发布核验、页面检查和上线确认按 `skills/blog-release-gate/SKILL.md` 执行。
5. 若用户已经明确说过“不要自己跑 bundle / jekyll build”“本地没装 Ruby / Bundler / Jekyll”或表达同等约束，禁止再次尝试本地构建。此时只能明确说明“本地构建门禁未执行，原因是用户已禁用或环境缺失”，并改走文本核对、模板逻辑检查、CI 工作流说明或等待线上构建结果；不要再自行探测 `bundle`、`ruby`、`gem` 是否存在。

## 提交与 Pull Request 规范
- 提交信息遵循 Conventional Commits，例如：
  - `feat(theme): 重构为暗色阅读优先主题`
  - `fix(ci): ...`
  - `content(profile): 更新关于页与标签体系`
- 每次提交聚焦单一主题（主题样式、内容、CI、脚本分开提交）。
- PR 建议包含：
  - 变更内容与动机。
  - 受影响的页面或文件路径。
  - 涉及 UI 时附截图。
  - 本地构建或 CI 通过说明。

## Git 提交策略
- 提交格式使用 Conventional Commits。
- 允许执行：`git add`、`git commit`。
- 禁止执行：`git reset --hard`、`git checkout --`、`git revert`（除非用户明确要求）。
- 若工作区存在与当前任务无关的改动，不要回滚，跳过并只提交本次相关文件。
- commit格式为[ADD]中文博客内容
